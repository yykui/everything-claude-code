# 产品能力

此技能将产品意图转化为明确的工程约束条件。

当问题不是"我们应该构建什么？"而是"在实现开始之前必须确认哪些内容？"时，使用此技能。

## 何时使用

- 存在 PRD、路线图条目、讨论或创始人备注，但实现约束仍然隐含时
- 功能跨越多个服务、代码库或团队，编码前需要能力契约时
- 产品意图清晰，但架构、数据、生命周期或策略影响仍不明确时
- 高级工程师在评审中反复重申相同的隐性假设时
- 需要一个可在多个工作台和会话中复用的制品时

## 规范制品

如果代码库中存在持久化的产品上下文文件，如 `PRODUCT.md`、`docs/product/` 或程序规格目录，请在其中更新。

如果尚不存在能力清单，请使用以下模板创建：

- `docs/examples/product-capability-template.md`

目标不是创建另一个规划栈，而是让隐性能力约束变得持久且可复用。

## 不可妥协的规则

- 不得捏造产品事实。明确标记未解决的问题。
- 将用户可见的承诺与实现细节分开。
- 明确指出哪些是固定策略、哪些是架构偏好、哪些仍待讨论。
- 如果请求与现有代码库约束冲突，请直接说明，而不是回避。
- 优先使用一个可复用的能力制品，而非分散的临时笔记。

## 输入

只读取必要内容：

1. 产品意图
   - issue、讨论、PRD、路线图备注、创始人消息
2. 当前架构
   - 相关代码库文档、契约、Schema、路由、现有工作流
3. 现有能力上下文
   - `PRODUCT.md`、设计文档、RFC、迁移说明、运营模型文档
4. 交付约束
   - 认证、计费、合规、发布、向后兼容性、性能、评审策略

## 核心工作流

### 1. 重述能力

将需求压缩为一句精确的陈述：

- 用户或操作者是谁
- 此功能发布后存在哪些新能力
- 因此发生了哪些结果变化

如果这句陈述薄弱，实现将会偏离。

### 2. 解析能力约束

提取实现前必须满足的约束条件：

- 业务规则
- 范围边界
- 不变量
- 信任边界
- 数据所有权
- 生命周期转换
- 发布/迁移要求
- 失败与恢复预期

这些内容往往只存在于高级工程师的记忆中。

### 3. 定义面向实现的契约

生成 SRS 风格的能力计划，包含：

- 能力摘要
- 明确的非目标
- 参与者与界面
- 必要的状态与转换
- 接口 / 输入 / 输出
- 数据模型影响
- 安全 / 计费 / 策略约束
- 可观测性与操作者要求
- 阻塞实现的未决问题

### 4. 转化为执行

以明确的交接结束：

- 可直接实现
- 需要先进行架构评审
- 需要先进行产品澄清

如有必要，指向下一个 ECC 原生通道：

- `project-flow-ops`
- `workspace-surface-audit`
- `api-connector-builder`
- `dashboard-builder`
- `tdd-workflow`
- `verification-loop`

## 输出格式

按以下顺序返回结果：

```text
CAPABILITY
- 一段话重述

CONSTRAINTS
- 固定规则、不变量和边界

IMPLEMENTATION CONTRACT
- 参与者
- 界面
- 状态与转换
- 接口/数据影响

NON-GOALS
- 此通道明确不负责的内容

OPEN QUESTIONS
- 仍需解答的阻塞问题或产品决策

HANDOFF
- 下一步应做什么以及应由哪个 ECC 通道承接
```

## 良好结果

- 产品意图现在足够具体，可以直接实现，无需在 PR 中途重新发现隐性约束。
- 工程评审有一个持久制品，而不必依赖记忆或 Slack 上下文。
- 生成的计划可在 Claude Code、Codex、Cursor、OpenCode 和 ECC 2.0 规划界面中复用。
