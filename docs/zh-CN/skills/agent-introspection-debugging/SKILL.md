# 代理内省调试

当代理运行反复失败、消耗 token 却毫无进展、循环调用相同工具，或偏离预定任务时，请使用本技能。

这是一个工作流技能，而非隐式运行时。它教导代理在上报给人类之前先系统地自我调试。

## 何时激活

- 达到最大工具调用次数或循环限制失败
- 反复重试但没有任何实质进展
- 上下文增长或提示词漂移导致输出质量开始下降
- 文件系统或环境状态与预期不符
- 工具故障可通过诊断和较小的纠正措施恢复

## 适用范围

在以下情况下激活本技能：
- 在盲目重试之前捕获故障状态
- 诊断常见的代理特定故障模式
- 执行受限恢复操作
- 生成结构化的人类可读调试报告

以下情况请勿使用本技能作为主要来源：
- 代码变更后的功能验证；请使用 `verification-loop`
- 当存在更具针对性的 ECC 技能时，针对特定框架的调试
- 当前运行框架无法自动执行的运行时承诺

## 四阶段循环

### 阶段一：故障捕获

在尝试恢复之前，精确记录故障。

捕获内容：
- 错误类型、错误信息及可用的堆栈跟踪
- 最后一次有意义的工具调用序列
- 代理当时试图做什么
- 当前上下文压力：重复提示词、过大的粘贴日志、重复计划或失控的笔记
- 当前环境假设：工作目录、分支、相关服务状态、预期文件

最低捕获模板：

```markdown
## 故障捕获
- 会话/任务：
- 进行中的目标：
- 错误：
- 最后成功的步骤：
- 最后失败的工具/命令：
- 发现的重复模式：
- 需要验证的环境假设：
```

### 阶段二：根因诊断

在做任何更改之前，将故障与已知模式匹配。

| 模式 | 可能原因 | 检查方法 |
| --- | --- | --- |
| 达到最大工具调用次数/重复相同命令 | 循环或无退出的观察路径 | 检查最近 N 次工具调用的重复情况 |
| 上下文溢出/推理质量下降 | 无限制的笔记、重复计划、过大日志 | 检查近期上下文中的重复内容和低信噪比内容 |
| `ECONNREFUSED`/超时 | 服务不可用或端口错误 | 验证服务健康状态、URL 和端口假设 |
| `429`/配额耗尽 | 重试风暴或缺少退避机制 | 统计重复调用次数并检查重试间隔 |
| 写入后文件缺失/差异过期 | 竞争条件、工作目录错误或分支漂移 | 重新检查路径、工作目录、git 状态和实际文件是否存在 |
| "修复"后测试仍失败 | 假设错误 | 隔离具体失败的测试并重新推导问题 |

诊断问题：
- 这是逻辑故障、状态故障、环境故障还是策略故障？
- 代理是否已偏离真实目标，开始优化错误的子任务？
- 故障是确定性的还是偶发性的？
- 能验证该诊断的最小可逆操作是什么？

### 阶段三：受限恢复

用改变诊断范围所需的最小操作来恢复。

安全恢复操作：
- 停止重复重试，重新陈述假设
- 裁剪低信噪比上下文，只保留当前目标、阻塞因素和证据
- 重新检查实际的文件系统/分支/进程状态
- 将任务缩小到一个失败命令、一个文件或一个测试
- 从推测性推理切换到直接观察
- 当故障高风险或受外部阻塞时，上报给人类

除非通过当前环境中的真实工具实际执行，否则不要声称进行了不受支持的自动修复操作，例如"重置代理状态"或"更新运行框架配置"。

受限恢复检查清单：

```markdown
## 恢复操作
- 选定的诊断：
- 执行的最小操作：
- 此操作安全的原因：
- 证明修复有效的证据是什么：
```

### 阶段四：内省报告

以一份报告作为结束，使下一个代理或人类能够清晰理解恢复过程。

```markdown
## 代理自我调试报告
- 会话/任务：
- 故障：
- 根因：
- 恢复操作：
- 结果：成功 | 部分 | 受阻
- Token/时间消耗风险：
- 需要后续跟进：
- 后续需编码的预防性变更：
```

## 恢复启发式规则

按以下顺序优先执行这些干预措施：

1. 用一句话重新陈述真实目标。
2. 验证实际状态，而非依赖记忆。
3. 缩小失败范围。
4. 执行一次区分性检查。
5. 只有在此之后才重试。

不良模式：
- 用略微不同的措辞重复相同操作三次

良好模式：
- 捕获故障
- 对模式进行分类
- 执行一次直接检查
- 仅当检查支持时才更改计划

## 与 ECC 的集成

- 如果代码已更改，恢复后使用 `verification-loop`。
- 当故障模式值得转化为直觉或后续技能时，使用 `continuous-learning-v2`。
- 当问题不是技术故障而是决策模糊时，使用 `council`。
- 当故障来自冲突的本地状态或仓库漂移时，使用 `workspace-surface-audit`。

## 输出标准

当本技能激活时，不要仅以"我修复了"结尾。

始终提供：
- 故障模式
- 根因假设
- 恢复操作
- 证明情况已改善或仍受阻的证据
