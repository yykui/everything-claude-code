# 智能体内省调试

当智能体运行反复失败、消耗令牌却没有进展、在相同工具上循环，或偏离预期任务时，使用此技能。

这是一个工作流技能，不是隐藏的运行时。它教会智能体在升级到人工干预之前系统地自我调试。

## 何时激活

- 达到最大工具调用/循环次数限制失败
- 反复重试却没有实质性进展
- 上下文增长或提示漂移开始降低输出质量
- 文件系统或环境状态与预期不符
- 工具失败可通过诊断和较小的纠正措施恢复

## 范围边界

激活此技能用于：
- 在盲目重试之前捕获失败状态
- 诊断常见的智能体特定失败模式
- 应用受控恢复措施
- 生成结构化的人类可读调试报告

不要将此技能作为以下场景的主要来源：
- 代码更改后的功能验证；请使用 `verification-loop`
- 当更窄的 ECC 技能已经存在时的框架特定调试
- 当前运行时无法自动强制执行的运行时承诺

## 四阶段循环

### 第一阶段：失败捕获

在尝试恢复之前，精确记录失败情况。

捕获内容：
- 错误类型、消息和堆栈跟踪（如果可用）
- 最后一次有意义的工具调用序列
- 智能体尝试做什么
- 当前上下文压力：重复提示、过大的粘贴日志、重复计划或失控的注释
- 当前环境假设：工作目录、分支、相关服务状态、预期文件

最小捕获模板：

```markdown
## Failure Capture
- Session / task:
- Goal in progress:
- Error:
- Last successful step:
- Last failed tool / command:
- Repeated pattern seen:
- Environment assumptions to verify:
```

### 第二阶段：根本原因诊断

在更改任何内容之前，将失败与已知模式匹配。

| 模式 | 可能原因 | 检查方式 |
| --- | --- | --- |
| 最大工具调用/反复执行相同命令 | 循环或无退出观察路径 | 检查最后 N 次工具调用中的重复 |
| 上下文溢出/推理能力下降 | 无界注释、重复计划、过大日志 | 检查近期上下文中的重复和低信号内容 |
| `ECONNREFUSED` / 超时 | 服务不可用或端口错误 | 验证服务健康状态、URL 和端口假设 |
| `429` / 配额耗尽 | 重试风暴或缺少退避 | 统计重复调用并检查重试间隔 |
| 写入后文件丢失/旧差异 | 竞态、错误工作目录或分支漂移 | 重新检查路径、工作目录、git 状态和实际文件存在性 |
| "修复"后测试仍然失败 | 假设错误 | 隔离确切的失败测试并重新推导 bug |

诊断问题：
- 这是逻辑失败、状态失败、环境失败还是策略失败？
- 智能体是否失去了真实目标并开始优化错误的子任务？
- 失败是确定性的还是瞬时性的？
- 能验证诊断的最小可逆操作是什么？

### 第三阶段：受控恢复

用改变诊断表面的最小操作进行恢复。

安全恢复措施：
- 停止重复重试并重新陈述假设
- 精简低信号上下文，只保留当前目标、阻塞因素和证据
- 重新检查实际文件系统/分支/进程状态
- 将任务缩小到一个失败命令、一个文件或一个测试
- 从推测性推理切换到直接观察
- 当失败是高风险或外部阻塞时，升级给人工处理

不要声明不受支持的自动修复操作，如"重置智能体状态"或"更新运行时配置"，除非你实际上在当前环境中通过真实工具执行这些操作。

受控恢复检查清单：

```markdown
## Recovery Action
- Diagnosis chosen:
- Smallest action taken:
- Why this is safe:
- What evidence would prove the fix worked:
```

### 第四阶段：内省报告

以一份使恢复过程对下一个智能体或人类清晰可读的报告作为结束。

```markdown
## Agent Self-Debug Report
- Session / task:
- Failure:
- Root cause:
- Recovery action:
- Result: success | partial | blocked
- Token / time burn risk:
- Follow-up needed:
- Preventive change to encode later:
```

## 恢复启发式方法

按顺序优先采用以下干预措施：

1. 用一句话重新陈述真实目标。
2. 验证世界状态而非依赖记忆。
3. 缩小失败范围。
4. 执行一次有区分性的检查。
5. 只有这样才重试。

错误模式：
- 用略微不同的措辞三次重试相同操作

正确模式：
- 捕获失败
- 分类模式
- 执行一次直接检查
- 只有在检查支持时才更改计划

## 与 ECC 的集成

- 如果更改了代码，在恢复后使用 `verification-loop`。
- 当失败模式值得转化为本能或后续技能时，使用 `continuous-learning-v2`。
- 当问题不是技术失败而是决策模糊时，使用 `council`。
- 如果失败来自本地状态冲突或仓库漂移，使用 `workspace-surface-audit`。

## 输出标准

当此技能激活时，不要只以"我修复了它"结束。

始终提供：
- 失败模式
- 根本原因假设
- 恢复措施
- 情况现已好转或仍被阻塞的证据
