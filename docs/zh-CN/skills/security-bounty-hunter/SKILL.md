# 安全漏洞猎手

当目标是为负责任披露或漏洞赏金提交实施漏洞发现时使用此技能，而非宽泛的最佳实践审查。

## 何时使用

- 扫描代码库中的可利用漏洞
- 准备 Huntr、HackerOne 或类似平台的漏洞提交
- 分类时的核心问题是"这实际上值钱吗？"而非"这理论上不安全吗？"

## 工作原理

偏向远程可达、用户可控的攻击路径，丢弃平台通常拒绝的信息性或超出范围的发现。

## 在范围内的模式

以下类型的问题始终值得关注：

| 模式 | CWE | 典型影响 |
|------|-----|---------|
| 通过用户控制 URL 的 SSRF | CWE-918 | 内部网络访问、云元数据盗取 |
| 中间件或 API 守卫中的认证绕过 | CWE-287 | 未授权账户或数据访问 |
| 远程反序列化或上传到 RCE 路径 | CWE-502 | 代码执行 |
| 可达端点中的 SQL 注入 | CWE-89 | 数据泄露、认证绕过、数据破坏 |
| 请求处理器中的命令注入 | CWE-78 | 代码执行 |
| 文件服务路径中的路径遍历 | CWE-22 | 任意文件读写 |
| 自动触发的 XSS | CWE-79 | 会话盗取、管理员账户入侵 |

## 跳过这些

以下通常是低信号或超出赏金范围（除非项目说明另有规定）：

- 无远程路径的纯本地 `pickle.loads`、`torch.load` 或等效操作
- 仅 CLI 工具中的 `eval()` 或 `exec()`
- 对完全硬编码命令使用 `shell=True`
- 单独的缺失安全标头
- 没有利用影响的通用限流投诉
- 需要受害者手动粘贴代码的 Self-XSS
- 不在目标项目范围内的 CI/CD 注入
- 演示、示例或仅测试代码

## 工作流

1. 先检查范围：项目规则、SECURITY.md、披露渠道和排除项。
2. 找到真实入口点：HTTP 处理器、上传、后台任务、Webhook、解析器和集成端点。
3. 在有帮助的地方运行静态工具，但仅将其视为分类输入。
4. 端到端阅读真实代码路径。
5. 证明用户控制能够到达有意义的 sink。
6. 使用最小可行的安全 PoC 确认可利用性和影响。
7. 起草报告前检查是否存在重复。

## 示例分类循环

```bash
semgrep --config=auto --severity=ERROR --severity=WARNING --json
```

然后手动过滤：

- 去除测试、演示、fixtures、vendor 代码
- 去除纯本地或不可达路径
- 仅保留具有明确网络或用户可控路由的发现

## 报告结构

```markdown
## Description
[漏洞是什么以及为什么重要]

## Vulnerable Code
[文件路径、行范围和简短代码片段]

## Proof of Concept
[最小可行请求或脚本]

## Impact
[攻击者能够实现什么]

## Affected Version
[测试的版本、提交或部署目标]
```

## 质量门控

提交前确认：

- 代码路径可从真实用户或网络边界到达
- 输入确实由用户控制
- sink 有意义且可利用
- PoC 有效
- 该问题尚未被 advisory、CVE 或开放 ticket 覆盖
- 目标确实在赏金项目范围内
