# Agent Sort

当某个仓库需要项目专属的 ECC 表面，而非默认全量安装时，请使用本技能。

目标不是猜测什么"感觉有用"，而是从实际代码库中提取证据，对 ECC 组件进行分类。

## 何时使用

- 项目只需要 ECC 的子集，完整安装噪音过大
- 技术栈清晰，但没有人愿意逐一手动筛选技能
- 团队希望基于 grep 证据而非主观判断做出可重复的安装决策
- 需要将每日必加载的工作流表面与可搜索的库/参考表面分开
- 仓库的语言、规则或钩子集已漂移到错误状态，需要清理

## 不可违反的规则

- 以当前仓库作为事实来源，而非通用偏好
- 每个 DAILY 决策必须引用具体的仓库证据
- LIBRARY 不意味着"删除"；而是"保持可访问但默认不加载"
- 不安装当前仓库无法使用的钩子、规则或脚本
- 优先使用 ECC 原生表面；不引入第二套安装系统

## 输出物

按顺序生成以下产出物：

1. DAILY 清单
2. LIBRARY 清单
3. 安装计划
4. 验证报告
5. 可选的 `skill-library` 路由器（如果项目需要）

## 分类模型

仅使用两个桶：

- `DAILY`
  - 每次会话都应为该仓库加载
  - 与仓库的语言、框架、工作流或操作表面强匹配
- `LIBRARY`
  - 有保留价值，但默认不值得加载
  - 应通过搜索、路由技能或选择性手动使用保持可访问

## 证据来源

在做任何分类之前，使用仓库本地证据：

- 文件扩展名
- 包管理器和锁文件
- 框架配置
- CI 和钩子配置
- 构建/测试脚本
- 导入和依赖清单
- 明确描述技术栈的仓库文档

常用命令包括：

```bash
rg --files
rg -n "typescript|react|next|supabase|django|spring|flutter|swift"
cat package.json
cat pyproject.toml
cat Cargo.toml
cat pubspec.yaml
cat go.mod
```

## 并行审查遍历

如果可以使用并行子代理，将审查分成以下几轮：

1. 代理
   - 分类 `agents/*`
2. 技能
   - 分类 `skills/*`
3. 命令
   - 分类 `commands/*`
4. 规则
   - 分类 `rules/*`
5. 钩子和脚本
   - 分类钩子表面、MCP 健康检查、辅助脚本和 OS 兼容性
6. 扩展项
   - 分类上下文、示例、MCP 配置、模板和指导文档

如果子代理不可用，按顺序执行相同的遍历。

## 核心工作流

### 1. 读取仓库

在分类任何内容之前，先建立真实技术栈：

- 使用的语言
- 使用的框架
- 主要包管理器
- 测试技术栈
- 代码检查/格式化技术栈
- 部署/运行时表面
- 已有的操作集成

### 2. 构建证据表

对于每个候选表面，记录：

- 组件路径
- 组件类型
- 建议的桶
- 仓库证据
- 简短理由

使用以下格式：

```text
skills/frontend-patterns | skill | DAILY | 84 个 .tsx 文件，存在 next.config.ts | 核心前端技术栈
skills/django-patterns   | skill | LIBRARY | 无 .py 文件，无 pyproject.toml       | 此仓库未使用
rules/typescript/*       | rules | DAILY | package.json + tsconfig.json            | 活跃的 TS 仓库
rules/python/*           | rules | LIBRARY | 零 Python 源文件                      | 仅保持可访问
```

### 3. 决定 DAILY 与 LIBRARY

升级为 `DAILY` 的条件：

- 仓库明确使用对应技术栈
- 组件足够通用，对每次会话都有帮助
- 仓库已依赖对应的运行时或工作流

降级为 `LIBRARY` 的条件：

- 组件技术栈不匹配
- 仓库将来可能需要，但不是每天都需要
- 增加上下文开销却没有直接相关性

### 4. 构建安装计划

将分类转化为操作：

- DAILY 技能 -> 安装或保留在 `.claude/skills/`
- DAILY 命令 -> 仅在仍有用时保留为显式垫片
- DAILY 规则 -> 仅安装匹配的语言集
- DAILY 钩子/脚本 -> 仅保留兼容的
- LIBRARY 表面 -> 通过搜索或 `skill-library` 保持可访问

如果仓库已使用选择性安装，更新该计划而非创建另一套系统。

### 5. 创建可选的库路由器

如果项目需要可搜索的库表面，创建：

- `.claude/skills/skill-library/SKILL.md`

该路由器应包含：

- DAILY 与 LIBRARY 的简短说明
- 分组的触发关键词
- 库引用的存放位置

不要在路由器中重复每个技能的完整内容。

### 6. 验证结果

应用计划后，验证：

- 每个 DAILY 文件都存在于预期位置
- 过期的语言规则未被遗留为活跃状态
- 不兼容的钩子未被安装
- 最终安装实际与仓库技术栈匹配

返回一份紧凑报告，包含：

- DAILY 数量
- LIBRARY 数量
- 已移除的过期表面
- 待解决的问题

## 输出格式

按以下顺序返回结果：

```text
技术栈
- 语言/框架/运行时摘要

DAILY
- 始终加载的项目及其证据

LIBRARY
- 可搜索/参考项目及其证据

安装计划
- 应安装、移除或路由的内容

验证
- 已执行的检查和剩余缺口
```
