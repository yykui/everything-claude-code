# ECC Tools 费用审查

当用户怀疑 ECC Tools GitHub App 存在费用超支、过度创建 PR、绕过使用限制或将免费用户导入高级分析路径时，使用此技能。

本技能是针对兄弟仓库 [ECC-Tools](../../ECC-Tools) 的专项运营工作流，不是通用计费技能，也不是全仓库代码审查。

## 技能栈

根据需要将以下 ECC 原生技能引入工作流：

- `autonomous-loops`：用于跨越 webhook、队列、计费和重试的有界多步骤审查
- `agentic-engineering`：用于将请求路径拆解为离散的、可证明的单元
- `customer-billing-ops`：当仓库行为与客户影响计算需要明确分离时使用
- `search-first`：在发明辅助函数或重新实现仓库本地工具之前使用
- `security-review`：当涉及身份验证、使用门控、权限或密钥时使用
- `verification-loop`：用于证明重新运行的安全性和修复后的精确状态
- `tdd-workflow`：当修复需要在工作线程、路由器或计费路径中添加回归测试时使用

## 何时使用

- 用户提到 ECC Tools 费用激增、PR 递归、过度创建 PR、使用限制绕过或高级模型费用泄漏
- 任务在兄弟仓库 `ECC-Tools` 中，且依赖 webhook 处理器、队列工作线程、使用量预留、PR 创建逻辑或付费门控
- 客户反馈 App 创建了过多 PR、计费不正确，或分析代码后未产生可用结果

## 范围限制

- 在兄弟仓库 `ECC-Tools` 中工作，而非 `everything-claude-code`
- 除非用户明确要求修复，否则以只读模式开始
- 追踪分析费用时，不要修改无关的计费、结账或 UI 流程
- 将 App 生成的分支和 App 生成的 PR 视为递归风险的红色警报，除非已证明无误
- 明确区分以下三点：
  - 仓库侧的费用根本原因
  - 面向客户的计费影响
  - 需要待办跟进的产品或权限缺口

## 工作流

### 1. 冻结仓库范围

- 切换到兄弟仓库 `ECC-Tools`
- 首先检查分支和本地差异
- 确定审查的确切范围：
  - webhook 路由器
  - 队列生产者
  - 队列消费者
  - PR 创建路径
  - 使用量预留/计费路径
  - 模型路由路径

### 2. 在推测之前追踪入口

- 首先检查 `src/index.*` 或主入口点
- 在建议修复之前，映射所有入队路径
- 确认哪些 GitHub 事件共享同一队列类型
- 确认 push、pull_request、synchronize、comment 或手动重新运行事件是否可能汇聚到同一个高成本路径

### 3. 追踪工作线程和副作用

- 检查处理分析的队列消费者或定时工作线程
- 确认已入队的分析是否总以以下方式结束：
  - PR 创建
  - 分支创建
  - 文件更新
  - 调用高级模型
  - 使用量计数递增
- 如果分析可能消耗 token 后在输出持久化前失败，将其归类为"有费用无产出"的浪费

### 4. 审查高信号费用路径

#### PR 数量倍增

- 检查 PR 辅助函数和分支命名
- 检查去重逻辑、synchronize 事件处理以及对已有 PR 的复用
- 如果 App 生成的分支可以重新进入分析流程，将其视为优先级 0 的递归风险

#### 配额绕过

- 检查配额检查的位置与使用量预留或递增的位置
- 如果配额在入队前检查但使用量在工作线程内计费，则将并发前门请求视为真实的竞争条件

#### 高级模型费用泄漏

- 检查模型选择、层级分支和提供商路由
- 验证当高级密钥存在时，免费或受限用户是否仍能访问高级分析器

#### 重试浪费

- 检查重试循环、重复队列任务和确定性失败的重新运行
- 如果同一个非瞬态错误可以多次触发分析，在优化质量之前先修复这个问题

### 5. 按费用优先级排序修复

如果用户要求代码修改，按以下顺序优先修复：

1. 停止自动 PR 数量倍增
2. 停止配额绕过
3. 停止高级模型费用泄漏
4. 停止重复任务扇出和无意义的重试
5. 关闭重新运行/更新安全性缺口

除非同一根本原因明显跨越多个文件，否则将修复范围限制在一到三个直接修复内。

### 6. 使用最小化的验证步骤进行核实

- 仅重新运行涵盖已修改路径的目标测试或集成切片
- 验证费用路径是否现在：
  - 已被阻断
  - 已去重
  - 已降级为更低成本的分析
  - 或被提前拒绝
- 精确说明最终状态：
  - 已在本地修改
  - 已在本地验证
  - 已推送
  - 已部署
  - 仍处于阻断状态

## 高信号失败模式

### 1. 所有触发器共用一种队列类型

如果 push、PR 同步和手动审查都入队同一种任务，且工作线程总是创建 PR，则分析等于 PR 垃圾。

### 2. 入队后才预留使用量

如果在前门检查使用量，但只在工作线程内递增，则并发请求都能通过门控并超出配额。

### 3. 免费层进入高级路径

如果免费排队的任务在高级密钥存在时仍能路由到 Anthropic 或其他高级提供商，即使用户从未看到高级结果，这也是真实的费用泄漏。

### 4. App 生成的分支重新进入 webhook

如果 `pull_request.synchronize`、分支推送或评论触发的运行会在 App 拥有的分支上触发，App 可能会递归地分析自己的输出。

### 5. 持久化安全保障前的昂贵操作

如果系统可能消耗 token 后在 PR 创建、文件更新或分支冲突时失败，则在不产出价值的情况下浪费了费用。

## 注意事项

- 不要从广泛的仓库漫游开始；先理清 webhook -> 队列 -> 工作线程
- 不要将客户计费推断与代码支撑的产品事实混淆
- 在控制最高费用路径之前，不要修复低价值的质量问题
- 在重新运行窄范围验证步骤之前，不要声称费用已修复
- 除非用户要求，否则不要推送或部署
- 如果仓库本地已有正在进行的修改，不要触碰无关内容

## 验证标准

- 根本原因引用精确的文件路径和代码区域
- 修复按费用影响排序，而非代码整洁度
- 列出验证命令
- 最终状态区分本地修改、验证、推送和部署
