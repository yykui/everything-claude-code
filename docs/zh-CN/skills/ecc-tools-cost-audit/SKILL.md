# ECC Tools 费用审计

当用户怀疑 ECC Tools GitHub App 产生异常费用、过度创建 PR、绕过使用限制，或将免费用户路由到高级分析路径时，使用此技能。

这是针对同级仓库 [ECC-Tools](../../ECC-Tools) 的专项运营工作流，而非通用计费技能，也不是全仓库代码审查。

## 技能栈

在工作流中按需引入以下 ECC 原生技能：

- `autonomous-loops`：用于跨 webhook、队列、计费和重试的有界多步审计
- `agentic-engineering`：将请求路径拆解为离散、可证明的单元
- `customer-billing-ops`：当仓库行为与客户影响计算需要清晰分离时
- `search-first`：在创造辅助函数或重新实现仓库本地工具之前
- `security-review`：涉及身份验证、使用门控、权限或密钥时
- `verification-loop`：用于证明重运行安全性和修复后的精确状态
- `tdd-workflow`：修复需要在 worker、路由器或计费路径中进行回归测试时

## 何时使用

- 用户提到 ECC Tools 消耗率、PR 递归、PR 过度创建、使用限制绕过或高级模型泄漏
- 任务在同级 `ECC-Tools` 仓库中，且依赖 webhook 处理器、队列 worker、使用预留、PR 创建逻辑或付费门控
- 客户反馈 App 创建了过多 PR、计费不正确，或分析代码后未产出可用结果

## 范围约束

- 在同级 `ECC-Tools` 仓库中操作，而非 `everything-claude-code`
- 除非用户明确要求修复，否则从只读模式开始
- 追踪分析消耗时，不要修改无关的计费、结账或 UI 流程
- 将 App 生成的分支和 PR 视为高危递归路径，直到另有证明
- 明确区分以下三点：
  - 仓库端的消耗根因
  - 客户端的计费影响
  - 需要进入待办的产品或权限缺口

## 工作流

### 1. 确定仓库范围

- 切换到同级 `ECC-Tools` 仓库
- 先检查分支和本地变更
- 确定审计的具体目标：
  - webhook 路由器
  - 队列生产者
  - 队列消费者
  - PR 创建路径
  - 使用预留/计费路径
  - 模型路由路径

### 2. 先追踪入口再提出假设

- 首先检查 `src/index.*` 或主入口文件
- 在建议修复前，映射所有入队路径
- 确认哪些 GitHub 事件共享同一队列类型
- 确认 push、pull_request、synchronize、comment 或手动重运行事件是否会汇聚到同一个高消耗路径

### 3. 追踪 worker 和副作用

- 检查处理分析的队列消费者或定时 worker
- 确认入队分析是否总会导致：
  - PR 创建
  - 分支创建
  - 文件更新
  - 高级模型调用
  - 使用量递增
- 若分析可消耗 token 但在输出持久化前失败，将其归类为"有消耗无输出"

### 4. 审计高信号消耗路径

#### PR 倍增

- 检查 PR 辅助函数和分支命名
- 检查去重、synchronize 事件处理和现有 PR 复用
- 若 App 生成的分支可重新进入分析流程，将其视为优先级 0 的递归风险

#### 配额绕过

- 检查配额在何处被检查，以及使用量在何处被预留或递增
- 若配额在入队前检查但使用量只在 worker 内部计费，将并发前门通过视为真实竞争条件

#### 高级模型泄漏

- 检查模型选择、套餐分支和提供商路由
- 验证免费或受限用户在存在高级密钥时是否仍可访问高级分析器

#### 重试消耗

- 检查重试循环、重复队列任务和确定性失败的重运行
- 若同一非瞬态错误可重复触发分析消耗，在质量改进之前先修复该问题

### 5. 按消耗优先级修复

若用户要求代码变更，按以下顺序排列修复优先级：

1. 停止自动 PR 倍增
2. 停止配额绕过
3. 停止高级模型泄漏
4. 停止重复任务扇出和无意义重试
5. 修复重运行/更新安全漏洞

将每次修复范围限制在一到三个直接修改，除非同一根因明显跨越多个文件。

### 6. 使用最小证明步骤进行验证

- 仅重运行覆盖变更路径的针对性测试或集成切片
- 验证消耗路径现在是否已：
  - 被阻断
  - 被去重
  - 降级为更低成本的分析
  - 或提前被拒绝
- 精确描述最终状态：
  - 本地已更改
  - 本地已验证
  - 已推送
  - 已部署
  - 仍被阻断

## 高信号失败模式

### 1. 所有触发器共用同一队列类型

若 push、PR 同步和手动审计都入队同一任务，且 worker 总是创建 PR，则分析等于 PR 刷屏。

### 2. 入队后才预留使用量

若使用量在前门检查但只在 worker 内递增，并发请求可全部通过门控并超出配额。

### 3. 免费套餐进入高级路径

若免费入队任务在存在密钥时仍可路由到 Anthropic 或其他高级提供商，即使用户从未看到高级结果，这也是真实的费用泄漏。

### 4. App 生成的分支重新进入 webhook

若 `pull_request.synchronize`、分支推送或评论触发的运行会在 App 拥有的分支上触发，App 可能会递归分析自身的输出。

### 5. 持久化安全之前执行高消耗操作

若系统可以消耗 token 后在 PR 创建、文件更新或分支冲突上失败，则在没有产出价值的情况下消耗了费用。

## 注意事项

- 不要一开始就对仓库进行宽泛探索；先确定 webhook -> 队列 -> worker 的链路
- 不要将客户计费推断与有代码支撑的产品事实混为一谈
- 不要在最高消耗路径得到控制之前修复低价值的质量问题
- 不要在未重运行针对性证明步骤的情况下声称消耗已修复
- 除非用户要求，不要推送或部署
- 如果仓库中已有进行中的无关本地变更，不要触碰它们

## 验证

- 根因引用精确的文件路径和代码区域
- 修复按消耗影响排序，而非代码整洁度
- 命名所用的证明命令
- 最终状态区分本地更改、验证、推送和部署
