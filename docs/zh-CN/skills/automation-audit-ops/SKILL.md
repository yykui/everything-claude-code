# 自动化审计运维

当用户询问哪些自动化处于活跃状态、哪些任务已损坏、存在哪些重叠，或哪些工具和连接器实际上正在发挥作用时，请使用本技能。

这是一个审计优先的运维技能。目标是在重写任何内容之前，生成有证据支撑的清单以及保留/合并/裁减/下一步修复的建议集。

## 技能栈

在工作流中需要时，引入这些 ECC 原生技能：

- `workspace-surface-audit`：用于连接器、MCP、钩子和应用清单
- `knowledge-ops`：当审计需要将实时仓库事实与持久上下文协调时
- `github-ops`：当答案依赖于 CI、定时工作流、Issues 或 PR 自动化时
- `ecc-tools-cost-audit`：当真正的问题是兄弟应用仓库中的 Webhook 扇出、队列任务或账单消耗时
- `research-ops`：当本地清单需要与当前平台支持或公开文档进行比较时
- `verification-loop`：用于证明修复后的状态，而非依赖假定的恢复

## 何时使用

- 用户询问"我有哪些自动化"、"什么是活跃的"、"什么是损坏的"或"哪些有重叠"
- 任务跨越定时任务、GitHub Actions、本地钩子、MCP 服务器、连接器、包装器或应用集成
- 用户想知道什么内容从另一个代理系统移植过来，什么还需要在 ECC 内重建
- 工作空间积累了多种完成同一件事的方式，用户想要一条规范路径

## 使用规范

- 除非用户明确要求修复，否则以只读方式开始
- 分别区分：
  - 已配置
  - 已认证
  - 近期已验证
  - 过期或损坏
  - 完全缺失
- 不要仅因技能或配置引用了某个工具就声称它处于活跃状态
- 在证据表存在之前，不要合并或删除重叠的表面

## 工作流

### 1. 清点真实表面

在进行推测之前，先读取当前的活跃表面：

- 仓库钩子和本地钩子脚本
- GitHub Actions 和定时工作流
- MCP 配置和已启用的服务器
- 基于连接器或应用的集成
- 包装脚本和仓库专属自动化入口点

按表面分组：

- 本地运行时
- 仓库 CI/自动化
- 已连接的外部系统
- 消息/通知
- 账单/客户运营
- 研究/监控

### 2. 按活跃状态对每项分类

对每个已发现的自动化，标记：

- 已配置
- 已认证
- 近期已验证
- 过期或损坏
- 缺失

然后对问题类型进行分类：

- 活跃故障
- 认证中断
- 过期状态
- 重叠或冗余
- 缺失能力

### 3. 追踪证明路径

用具体来源支撑每个重要声明：

- 文件路径
- 工作流运行记录
- 钩子日志
- 配置条目
- 近期命令输出
- 精确的失败特征

如果当前状态不明确，直接说明，而非假装审计已完成。

### 4. 以保留/合并/裁减/下一步修复作为结尾

对每个重叠或可疑的表面，给出一个明确建议：

- 保留
- 合并
- 裁减
- 下一步修复

价值在于将嘈杂的自动化整合到一条规范的 ECC 路径中，而非保留每条历史路径。

## 输出格式

```text
当前表面
- 自动化
- 来源
- 活跃状态
- 证明

发现
- 活跃故障
- 重叠
- 过期状态
- 缺失能力

建议
- 保留
- 合并
- 裁减
- 下一步修复

下一步 ECC 操作
- 需要强化的确切技能/钩子/工作流/应用路径
```

## 注意事项

- 当可以读取活跃清单时，不要凭记忆回答
- 不要将"存在于配置中"视为"正常工作"
- 在指出损坏的高价值路径之前，不要先修复低价值的冗余问题
- 如果用户要求的是清单，不要将任务扩展为仓库重写

## 验证

- 重要声明引用活跃的证明路径
- 每个已发现的自动化都标有明确的活跃状态类别
- 最终建议区分了保留/合并/裁减/下一步修复
