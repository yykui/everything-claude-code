# 自动化审计运维

当用户询问哪些自动化正在运行、哪些作业已损坏、存在哪些重叠，或者哪些工具和连接器实际上正在做有用的工作时使用此技能。

这是一个审计优先的操作员技能。工作是在重写任何内容之前生成基于证据的清单以及保留/合并/删除/下一步修复的建议集。

## 技能栈

在相关时将以下 ECC 原生技能纳入工作流：

- `workspace-surface-audit`：用于连接器、MCP、钩子和应用清单
- `knowledge-ops`：当审计需要将实时仓库真相与持久上下文协调时
- `github-ops`：当答案取决于 CI、计划工作流、问题或 PR 自动化时
- `ecc-tools-cost-audit`：当真正的问题是 webhook 扇出、排队作业或兄弟应用仓库中的账单燃耗时
- `research-ops`：当本地清单必须与当前平台支持或公共文档进行比较时
- `verification-loop`：用于证明修复后状态，而非依赖假设的恢复

## 何时使用

- 用户询问"我有哪些自动化"、"什么是活跃的"、"什么是损坏的"或"什么有重叠"
- 任务跨越 cron 作业、GitHub Actions、本地钩子、MCP 服务器、连接器、包装器或应用集成
- 用户想知道哪些是从另一个智能体系统迁移过来的，哪些还需要在 ECC 内重建
- 工作区积累了多种实现相同功能的方式，用户想要一条规范通道

## 护栏

- 除非用户明确要求修复，否则从只读模式开始
- 区分：
  - 已配置
  - 已认证
  - 最近已验证
  - 过时或已损坏
  - 完全缺失
- 不要仅因为技能或配置引用了某个工具就声称它是活跃的
- 在证据表存在之前，不要合并或删除重叠界面

## 工作流

### 1. 清点真实界面

在理论化之前读取当前实时界面：

- 仓库钩子和本地钩子脚本
- GitHub Actions 和计划工作流
- MCP 配置和已启用服务器
- 连接器或应用支持的集成
- 包装脚本和仓库特定自动化入口点

按界面分组：

- 本地运行时
- 仓库 CI/自动化
- 已连接的外部系统
- 消息传递/通知
- 账单/客户运营
- 研究/监控

### 2. 按活跃状态对每个项目分类

对于每个已发现的自动化，标记：

- 已配置
- 已认证
- 最近已验证
- 过时或已损坏
- 缺失

然后分类问题类型：

- 活跃中断
- 认证中断
- 过时状态
- 重叠或冗余
- 缺失能力

### 3. 追踪证明路径

用具体来源支持每个重要声明：

- 文件路径
- 工作流运行
- 钩子日志
- 配置条目
- 近期命令输出
- 确切的失败签名

如果当前状态不明确，直接说明，而非假装审计已完成。

### 4. 以保留/合并/删除/下一步修复结束

对于每个重叠或可疑界面，返回一个决策：

- 保留
- 合并
- 删除
- 下一步修复

价值在于将嘈杂的自动化整合到一条规范的 ECC 通道中，而非保留每条历史路径。

## 输出格式

```text
CURRENT SURFACE
- automation
- source
- live state
- proof

FINDINGS
- active breakage
- overlap
- stale status
- missing capability

RECOMMENDATION
- keep
- merge
- cut
- fix next

NEXT ECC MOVE
- exact skill / hook / workflow / app lane to strengthen
```

## 常见陷阱

- 当可以读取实时清单时，不要从记忆中回答
- 不要将"存在于配置中"视为"正在运行"
- 在命名损坏的高信号路径之前，不要修复低价值冗余
- 如果用户首先要求清单，不要将任务扩展为仓库重写

## 验证

- 重要声明引用实时证明路径
- 每个已发现的自动化标有明确的活跃状态类别
- 最终建议区分保留/合并/删除/下一步修复
