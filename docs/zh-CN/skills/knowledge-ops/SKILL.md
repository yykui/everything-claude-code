# 知识操作

管理多层知识系统，用于跨多个存储库摄取、整理、同步和检索知识。

优先使用实时工作区模型：
- 代码工作内容保存在真实克隆的仓库中
- 活跃执行上下文保存在 GitHub、Linear 以及仓库本地的 working-context 文件中
- 面向人类的笔记可保存在非仓库的 context/archive 文件夹中
- 跨机器的持久内存属于知识库，而非影子仓库工作区

## 何时激活

- 用户希望将信息保存到知识库
- 将文档、对话或数据摄取到结构化存储
- 跨系统同步知识（本地文件、MCP 内存、Supabase、Git 仓库）
- 去重或整理现有知识
- 用户说"保存到知识库"、"同步知识"、"我对 X 了解什么"、"摄取这个"、"更新知识库"
- 任何超出简单内存回溯的知识管理任务

## 知识架构

### 第一层：活跃执行真实来源
- **来源：** GitHub 问题、PR、讨论、发布说明，Linear 问题/项目/文档
- **用途：** 当前工作的运营状态
- **规则：** 如果某内容影响活跃的工程计划、路线图、上线或发布，优先在此处更新

### 第二层：Claude Code 内存（快速访问）
- **路径：** `~/.claude/projects/*/memory/`
- **格式：** 带 frontmatter 的 Markdown 文件
- **类型：** 用户偏好、反馈、项目上下文、参考资料
- **用途：** 跨对话持久化的快速访问上下文
- **会话开始时自动加载**

### 第三层：MCP 内存服务器（结构化知识图谱）
- **访问：** MCP 内存工具（create_entities、create_relations、add_observations、search_nodes）
- **用途：** 跨所有存储内存的语义搜索、关系映射
- **具备可查询图结构的跨会话持久化**

### 第四层：知识库仓库 / 持久文档存储
- **用途：** 经过精心整理的持久笔记、会话导出、综合研究、运营内存、长篇文档
- **规则：** 当内容不属于仓库代码时，这是跨机器上下文的首选持久存储

### 第五层：外部数据存储（Supabase、PostgreSQL 等）
- **用途：** 结构化数据、大文档存储、全文搜索
- **适合：** 内存文件容纳不下的文档、需要 SQL 查询的数据

### 第六层：本地 context/archive 文件夹
- **用途：** 面向人类的笔记、存档的游戏计划、本地媒体整理、临时非代码文档
- **规则：** 可写入信息存储，但不作为影子代码工作区
- **不用于：** 活跃代码变更或应保存在上游的仓库真实内容

## 摄取工作流

当需要捕获新知识时：

### 1. 分类
这是哪种类型的知识？
- 业务决策 → 内存文件（项目类型）+ MCP 内存
- 活跃路线图 / 发布 / 实现状态 → 先更新 GitHub + Linear
- 个人偏好 → 内存文件（用户/反馈类型）
- 参考信息 → 内存文件（参考类型）+ MCP 内存
- 大型文档 → 外部数据存储 + 内存中的摘要
- 对话/会话 → 知识库仓库 + 内存中的简短摘要

### 2. 去重
检查该知识是否已存在：
- 搜索内存文件中的现有条目
- 用相关词查询 MCP 内存
- 在创建本地笔记之前，检查 GitHub 或 Linear 中是否已有该信息
- 不要创建重复内容，更新现有条目即可。

### 3. 存储
写入适当的存储层：
- 始终更新 Claude Code 内存以便快速访问
- 使用 MCP 内存实现语义可搜索性和关系映射
- 当信息改变活跃项目真实内容时，优先更新 GitHub / Linear
- 为持久长篇内容提交到知识库仓库

### 4. 索引
更新所有相关的索引或摘要文件。

## 同步操作

### 对话同步
定期将对话历史同步到知识库：
- 来源：Claude 会话文件、Codex 会话、其他代理会话
- 目标：知识库仓库
- 生成会话索引以便快速浏览
- 提交并推送

### 工作区状态同步
将重要的工作区配置和脚本镜像到知识库：
- 生成目录地图
- 提交前脱敏敏感配置
- 随时间跟踪变更
- 不要将知识库或 archive 文件夹视为实时代码工作区

### GitHub / Linear 同步
当信息影响活跃执行时：
- 更新相关的 GitHub 问题、PR、讨论、发布说明或路线图线程
- 当工作需要持久规划上下文时，将支撑文档附加到 Linear
- 只有在本地笔记仍有附加价值时，才在事后进行镜像

### 跨来源知识同步
将多个来源的知识汇集到一处：
- Claude/ChatGPT/Grok 对话导出
- 浏览器书签
- GitHub 活动事件
- 编写状态摘要，提交并推送

## 内存模式

```
# 短期：当前会话上下文
使用 TodoWrite 进行会话内任务跟踪

# 中期：项目内存文件
写入 ~/.claude/projects/*/memory/ 以实现跨会话回溯

# 长期：GitHub / Linear / 知识库
将活跃执行真实内容放入 GitHub + Linear
将持久的综合上下文放入知识库仓库

# 语义层：MCP 知识图谱
使用 mcp__memory__create_entities 存储永久结构化数据
使用 mcp__memory__create_relations 进行关系映射
使用 mcp__memory__add_observations 添加关于已知实体的新事实
使用 mcp__memory__search_nodes 查找现有知识
```

## 最佳实践

- 保持内存文件简洁。归档旧数据，而非让文件无限增长。
- 对所有知识文件使用 frontmatter（YAML）作为元数据。
- 存储前先去重。先搜索，再创建或更新。
- 每组事实优先保留一个权威来源，避免在本地笔记、仓库文件和跟踪文档之间建立并行副本。
- 提交到 Git 前脱敏敏感信息（API 密钥、密码）。
- 对知识文件使用一致的命名约定（lowercase-kebab-case）。
- 用主题/类别标记条目，便于检索。

## 质量门控

完成任何知识操作前：
- 未创建重复条目
- 任何 Git 跟踪文件中的敏感数据已脱敏
- 索引和摘要已更新
- 已为数据类型选择合适的存储层
- 已在相关处添加交叉引用
