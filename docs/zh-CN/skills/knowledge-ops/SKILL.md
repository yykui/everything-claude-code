# 知识操作

管理多层知识系统，实现跨多个存储库的知识摄入、整理、同步与检索。

优先采用实时工作空间模型：
- 代码工作内容存放于真实克隆的仓库中
- 活跃执行上下文存放于 GitHub、Linear 及仓库本地的 working-context 文件中
- 面向人类的宏观笔记可存放于非仓库的 context/archive 文件夹中
- 跨机器的持久化内存属于知识库，而非影子仓库工作区

## 何时激活

- 用户希望将信息保存到知识库
- 将文档、对话或数据摄入到结构化存储中
- 跨系统同步知识（本地文件、MCP 内存、Supabase、Git 仓库）
- 对现有知识进行去重或整理
- 用户说"保存到 KB"、"同步知识"、"我对 X 知道什么"、"摄入这个"、"更新知识库"
- 超出简单内存调用的任何知识管理任务

## 知识架构

### 第一层：活跃执行事实来源
- **来源：** GitHub 问题、PR、讨论、发布说明，Linear 问题/项目/文档
- **用途：** 工作的当前操作状态
- **规则：** 如果某内容影响活跃的工程计划、路线图、发布流程或版本，优先存放于此

### 第二层：Claude Code 内存（快速访问）
- **路径：** `~/.claude/projects/*/memory/`
- **格式：** 带 frontmatter 的 Markdown 文件
- **类型：** 用户偏好、反馈、项目上下文、参考资料
- **用途：** 跨对话持久化的快速访问上下文
- **在会话开始时自动加载**

### 第三层：MCP 内存服务器（结构化知识图谱）
- **访问方式：** MCP 内存工具（create_entities、create_relations、add_observations、search_nodes）
- **用途：** 跨所有存储内存的语义搜索、关系映射
- **具有可查询图结构的跨会话持久化**

### 第四层：知识库仓库 / 持久文档存储
- **用途：** 精心整理的持久笔记、会话导出、综合研究、操作员内存、长篇文档
- **规则：** 当内容不属于仓库代码时，这是跨机器上下文的首选持久存储

### 第五层：外部数据存储（Supabase、PostgreSQL 等）
- **用途：** 结构化数据、大型文档存储、全文搜索
- **适合：** 内存文件无法容纳的大型文档、需要 SQL 查询的数据

### 第六层：本地 context/archive 文件夹
- **用途：** 面向人类的笔记、归档的游戏计划、本地媒体整理、临时非代码文档
- **规则：** 可用于信息存储，但不作为影子代码工作区
- **禁止用于：** 应存放在上游的活跃代码变更或仓库事实

## 摄入工作流

当需要捕获新知识时：

### 1. 分类
这是哪种类型的知识？
- 业务决策 -> 内存文件（项目类型）+ MCP 内存
- 活跃路线图 / 发布 / 实现状态 -> 优先存入 GitHub + Linear
- 个人偏好 -> 内存文件（用户/反馈类型）
- 参考信息 -> 内存文件（参考类型）+ MCP 内存
- 大型文档 -> 外部数据存储 + 内存中的摘要
- 对话/会话 -> 知识库仓库 + 内存中的简短摘要

### 2. 去重
检查该知识是否已存在：
- 在内存文件中搜索现有条目
- 用相关词语查询 MCP 内存
- 在创建本地笔记前，检查信息是否已存在于 GitHub 或 Linear 中
- 不要创建重复项，而是更新现有条目。

### 3. 存储
写入适当的存储层：
- 始终更新 Claude Code 内存以便快速访问
- 使用 MCP 内存实现语义可搜索性和关系映射
- 当信息改变了活跃项目事实时，优先更新 GitHub / Linear
- 对于持久性长篇补充，提交到知识库仓库

### 4. 索引
更新所有相关的索引或摘要文件。

## 同步操作

### 对话同步
定期将对话历史同步到知识库：
- 来源：Claude 会话文件、Codex 会话、其他 Agent 会话
- 目标：知识库仓库
- 生成会话索引以便快速浏览
- 提交并推送

### 工作区状态同步
将重要的工作区配置和脚本镜像到知识库：
- 生成目录映射
- 提交前对敏感配置进行脱敏处理
- 随时间追踪变更
- 不要将知识库或归档文件夹作为实时代码工作区

### GitHub / Linear 同步
当信息影响活跃执行时：
- 更新相关的 GitHub 问题、PR、讨论、发布说明或路线图线程
- 当工作需要持久规划上下文时，将支持文档附加到 Linear
- 仅当本地笔记仍有附加价值时，事后进行镜像

### 跨来源知识同步
将来自多个来源的知识汇聚到一处：
- Claude/ChatGPT/Grok 对话导出
- 浏览器书签
- GitHub 活动事件
- 撰写状态摘要，提交并推送

## 内存模式

```
# 短期：当前会话上下文
使用 TodoWrite 进行会话内任务追踪

# 中期：项目内存文件
写入 ~/.claude/projects/*/memory/ 以实现跨会话召回

# 长期：GitHub / Linear / KB
将活跃执行事实存入 GitHub + Linear
将持久化综合上下文存入知识库仓库

# 语义层：MCP 知识图谱
使用 mcp__memory__create_entities 存储永久结构化数据
使用 mcp__memory__create_relations 进行关系映射
使用 mcp__memory__add_observations 为已知实体添加新事实
使用 mcp__memory__search_nodes 查找现有知识
```

## 最佳实践

- 保持内存文件简洁。归档旧数据，而不是让文件无限增长。
- 对所有知识文件使用 frontmatter（YAML）作为元数据。
- 存储前先去重。先搜索，再创建或更新。
- 每个事实集优先选择一个规范位置。避免同一计划在本地笔记、仓库文件和追踪器文档中并行存在多份副本。
- 提交到 Git 前对敏感信息（API 密钥、密码）进行脱敏处理。
- 对知识文件使用一致的命名规范（lowercase-kebab-case）。
- 为条目添加主题/类别标签以便检索。

## 质量关卡

在完成任何知识操作前：
- 未创建重复条目
- 所有 Git 追踪文件中的敏感数据已脱敏
- 索引和摘要已更新
- 已为数据类型选择了合适的存储层
- 在相关位置添加了交叉引用
