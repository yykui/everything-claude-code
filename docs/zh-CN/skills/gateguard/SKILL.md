# GateGuard — 强制事实的行动前门控

一个 PreToolUse 钩子，强制 Claude 在编辑前进行调查。它不是自我评估（"你确定吗？"），而是要求具体事实。调查行为本身创造了自我评估从未能创造的上下文感知。

## 何时激活

- 在文件编辑会影响多个模块的任何代码库中工作
- 具有特定模式或日期格式的数据文件项目
- AI 生成代码必须匹配现有模式的团队
- 任何 Claude 倾向于猜测而非调查的工作流

## 核心概念

LLM 自我评估不起作用。问"你是否违反了任何规则？"答案永远是"没有"。这已通过实验验证。

但问"列出导入此模块的每个文件"会强制 LLM 运行 Grep 和 Read。调查本身创造了改变输出的上下文。

**三阶段门控：**

```
1. 拒绝  — 阻断第一次 Edit/Write/Bash 尝试
2. 强制  — 告知模型需要收集哪些具体事实
3. 允许  — 在事实呈现后允许重试
```

没有竞品做到这全部三点。大多数止步于"拒绝"。

## 证据

两个独立的 A/B 测试，相同的代理，相同的任务：

| 任务 | 有门控 | 无门控 | 差距 |
| --- | --- | --- | --- |
| 分析模块 | 8.0/10 | 6.5/10 | +1.5 |
| Webhook 验证器 | 10.0/10 | 7.0/10 | +3.0 |
| **平均** | **9.0** | **6.75** | **+2.25** |

两个代理都能产出可运行且通过测试的代码。差异在于设计深度。

## 门控类型

### Edit / MultiEdit 门控（每个文件第一次编辑时触发）

MultiEdit 的处理方式相同——批次中的每个文件单独门控。

```
在编辑 {file_path} 之前，请呈现以下事实：

1. 列出所有导入/引用此文件的文件（使用 Grep）
2. 列出受此更改影响的公共函数/类
3. 如果此文件读写数据文件，请展示字段名、结构和日期格式
   （使用经过脱敏或合成的值，而非原始生产数据）
4. 逐字引用用户当前的指令
```

### Write 门控（第一次创建新文件时触发）

```
在创建 {file_path} 之前，请呈现以下事实：

1. 命名将调用此新文件的文件和行号
2. 确认没有现有文件服务于相同目的（使用 Glob）
3. 如果此文件读写数据文件，请展示字段名、结构和日期格式
   （使用经过脱敏或合成的值，而非原始生产数据）
4. 逐字引用用户当前的指令
```

### 破坏性 Bash 门控（每次破坏性命令时触发）

触发条件：`rm -rf`、`git reset --hard`、`git push --force`、`drop table` 等。

```
1. 列出此命令将修改或删除的所有文件/数据
2. 写出一行回滚方案
3. 逐字引用用户当前的指令
```

### 常规 Bash 门控（每次会话一次）

```
1. 用一句话描述当前的用户请求
2. 说明此具体命令验证或产出什么
```

## 快速开始

### 方式 A：使用 ECC 钩子（零安装）

`scripts/hooks/gateguard-fact-force.js` 处的钩子已包含在本插件中。通过 hooks.json 启用。

### 方式 B：带配置的完整软件包

```bash
pip install gateguard-ai
gateguard init
```

这会添加 `.gateguard.yml` 用于项目级配置（自定义消息、忽略路径、门控开关）。

## 反模式

- **不要用自我评估替代门控。** "你确定吗？"总会得到"是的"。这已通过实验验证。
- **不要跳过数据模式检查。** 两个 A/B 测试代理都假设使用 ISO-8601 日期，而真实数据使用 `%Y/%m/%d %H:%M`。检查数据结构（使用脱敏值）可以防止这整类错误。
- **不要对每一个 Bash 命令都进行门控。** 常规 bash 每次会话门控一次，破坏性 bash 每次都门控。这种平衡避免了速度损耗，同时捕获真实风险。

## 最佳实践

- 让门控自然触发。不要尝试预先回答门控问题——调查本身才是提升质量的关键。
- 为你的领域自定义门控消息。如果项目有特定约定，将其添加到门控提示中。
- 使用 `.gateguard.yml` 忽略 `.venv/`、`node_modules/`、`.git/` 等路径。

## 相关技能

- `safety-guard` — 运行时安全检查（互补而非重叠）
- `code-reviewer` — 编辑后审查（GateGuard 是编辑前调查）
