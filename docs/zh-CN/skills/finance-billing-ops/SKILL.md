# 财务计费运营

当用户需要了解资金状况、定价、退款、团队席位逻辑，或产品的实际行为是否与官网和销售文案相符时，使用本技能。

本技能比 `customer-billing-ops` 更宽泛。后者用于客户补救，本技能用于运营真相：收入状态、定价决策、团队计费以及代码支撑的计费行为。

## 技能栈

根据需要将以下 ECC 原生技能引入工作流：

- `customer-billing-ops`：用于特定客户的补救和跟进
- `research-ops`：当竞品定价或当前市场证据重要时使用
- `market-research`：当答案应以定价建议作为结论时使用
- `github-ops`：当计费事实依赖于兄弟仓库的代码、待办事项或发布状态时使用
- `verification-loop`：当答案需要证明结账、席位处理或权限行为时使用

## 何时使用

- 用户询问 Stripe 销售额、退款、MRR 或近期客户活动
- 用户询问团队计费、按席位计费或配额叠加在代码中是否真实存在
- 用户需要竞品定价比较或定价模型基准
- 问题混合了收入事实与产品实施真相

## 操作限制

- 区分实时数据和已保存的快照
- 明确分离：
  - 收入事实
  - 客户影响
  - 代码支撑的产品真相
  - 建议
- 除非实际权限路径强制执行，否则不得说"按席位"
- 不要假设重复订阅意味着重复价值

## 工作流

### 1. 从最新的计费证据开始

优先使用实时计费数据。如果数据非实时，明确说明快照时间戳。

梳理全貌：

- 已付款销售
- 活跃订阅
- 失败或未完成的结账
- 退款
- 争议
- 重复订阅

### 2. 将客户事件与产品真相分开

如果问题是客户特定的，先分类：

- 重复结账
- 真实的团队意图
- 损坏的自助服务控制
- 未满足的产品价值
- 支付失败或设置未完成

然后将其与更广泛的产品问题分开：

- 团队计费是否真实存在？
- 席位是否真正被计数？
- 结账数量是否影响权限？
- 官网是否夸大了当前行为？

### 3. 检查代码支撑的计费行为

如果答案取决于实施真相，检查代码路径：

- 结账
- 定价页面
- 权限计算
- 席位或配额处理
- 安装与用户使用量逻辑
- 计费门户或自助服务管理支持

### 4. 以决策和产品缺口作为结论

报告：

- 销售快照
- 问题诊断
- 产品真相
- 建议的运营行动
- 产品或待办缺口

## 输出格式

```text
快照
- 时间戳
- 收入/订阅/异常

客户影响
- 受影响的用户
- 发生了什么

产品真相
- 代码实际做了什么
- 官网或销售文案的声称

决策
- 退款 / 保留 / 转换 / 不操作

产品缺口
- 需要构建或修复的具体跟进事项
```

## 注意事项

- 不要将失败的尝试与净收入混淆
- 不要仅凭营销语言推断团队计费
- 当前证据可用时，不要凭记忆比较竞品定价
- 不要在分类问题之前直接从诊断跳到退款

## 验证标准

- 答案包含实时数据声明或快照时间戳
- 产品真相声明有代码支撑
- 客户影响与更广泛的定价/产品结论明确分离
