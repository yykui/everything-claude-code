# 财务计费运营

当用户想了解资金流向、定价、退款、团队席位逻辑，或产品实际行为是否与网站及销售文案一致时使用。

这比 `customer-billing-ops` 的范围更广。那个技能用于客户补救；本技能用于运营真相：收入状态、定价决策、团队计费，以及有代码支撑的计费行为。

## 技能栈

在工作流中按需引入以下 ECC 原生技能：

- `customer-billing-ops`：用于客户特定的补救和跟进
- `research-ops`：当竞品定价或当前市场证据很重要时
- `market-research`：当答案需要以定价建议收尾时
- `github-ops`：当计费真相依赖于同级仓库的代码、待办或发布状态时
- `verification-loop`：当答案需要证明结账、席位处理或权限行为时

## 何时使用

- 用户询问 Stripe 销售额、退款、MRR 或近期客户活动
- 用户询问代码中是否真正实现了团队计费、按席位计费或配额叠加
- 用户需要竞品定价比较或定价模型基准
- 问题将收入事实与产品实现真相混合在一起

## 操作规范

- 区分实时数据与保存的快照
- 明确区分：
  - 收入事实
  - 客户影响
  - 有代码支撑的产品真相
  - 建议
- 除非实际的权限路径强制执行，否则不要说"按席位"
- 不要假设重复订阅意味着重复价值

## 工作流

### 1. 从最新的计费证据开始

优先使用实时计费数据。若数据非实时，明确说明快照时间戳。

规范化全景：

- 已付款销售
- 活跃订阅
- 失败或未完成的结账
- 退款
- 争议
- 重复订阅

### 2. 区分客户事件与产品真相

若问题针对特定客户，先分类：

- 重复结账
- 真实团队意图
- 自助控制失效
- 产品价值未兑现
- 支付失败或设置不完整

然后将其与更广泛的产品问题分开：

- 团队计费是否真实存在？
- 席位是否实际被计数？
- 结账数量是否影响权限？
- 网站是否夸大了当前行为？

### 3. 检查有代码支撑的计费行为

若答案依赖于实现真相，检查代码路径：

- 结账
- 定价页
- 权限计算
- 席位或配额处理
- 安装与用户使用逻辑
- 计费门户或自助管理支持

### 4. 以决策和产品缺口结尾

报告：

- 销售快照
- 问题诊断
- 产品真相
- 推荐的运营行动
- 产品或待办缺口

## 输出格式

```text
SNAPSHOT
- timestamp
- revenue / subscriptions / anomalies

CUSTOMER IMPACT
- who is affected
- what happened

PRODUCT TRUTH
- what the code actually does
- what the website or sales copy claims

DECISION
- refund / preserve / convert / no-op

PRODUCT GAP
- exact follow-up item to build or fix
```

## 注意事项

- 不要将失败的尝试与净收入混为一谈
- 不要仅凭营销语言推断团队计费
- 当有当前证据可用时，不要凭记忆比较竞品定价
- 不要在未分类问题的情况下直接从诊断跳到退款

## 验证

- 答案包含实时数据声明或快照时间戳
- 产品真相声明有代码支撑
- 客户影响与更广泛的定价/产品结论被清晰分开
