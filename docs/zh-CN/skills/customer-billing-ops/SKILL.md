# 客户计费运营

此技能用于真实的客户操作，而非通用支付 API 设计。

目标是帮助运营人员回答：这位客户是谁、发生了什么、最安全的修复方案是什么，以及应该发送什么后续跟进？

## 何时使用

- 客户反映计费出现问题、要求退款或无法取消
- 调查重复订阅、意外扣费、续费失败或流失风险
- 审查计划组合、活跃订阅、年付与月付转换，或团队席位混淆
- 创建或验证计费门户流程
- 审核涉及订阅、发票、退款或支付方式的支持投诉

## 首选工具界面

- 首先使用 Stripe 等已连接的计费工具
- 仅将邮件、GitHub 或问题追踪器用作支持性证据
- 当平台已提供所需控制时，优先使用托管的计费/客户门户而非自定义账户管理代码

## 安全护栏

- 绝不在响应中暴露密钥、完整卡片详情或不必要的客户个人身份信息
- 不盲目退款；先对问题进行分类
- 区分以下情况：
  - 意外重复购买
  - 真实的多席位或团队购买
  - 产品故障 / 未兑现的价值
  - 失败或未完成的结账
  - 因缺少自助控制而取消
- 对于年度计划、团队计划和按比例计算的状态，在采取行动前验证合同形式

## 工作流

### 1. 清晰识别客户

从最强的标识符开始：

- 客户电子邮件
- Stripe 客户 ID
- 订阅 ID
- 发票 ID
- GitHub 用户名或支持邮件（若已知可映射回计费）

返回简洁的身份摘要：

- 客户信息
- 活跃订阅
- 已取消的订阅
- 发票
- 明显异常，如重复的活跃订阅

### 2. 对问题进行分类

在采取行动之前将案例归入一个类别：

| 案例 | 典型处理 |
|------|----------------|
| 重复的个人订阅 | 取消额外的，考虑退款 |
| 真实的多席位/团队意图 | 保留席位，明确计费模型 |
| 支付失败 / 结账未完成 | 通过门户恢复或更新支付方式 |
| 缺少自助控制 | 提供门户、取消路径或发票访问 |
| 产品故障或信任损失 | 退款、道歉、记录产品问题 |

### 3. 首先采取最安全的可逆操作

推荐顺序：

1. 恢复自助管理
2. 修复重复或损坏的计费状态
3. 仅退款受影响的费用或重复费用
4. 记录原因
5. 发送简短的客户后续跟进

若修复需要产品工作，分开处理：

- 立即进行客户补救
- 将产品 Bug / 工作流缺口列入待办事项

### 4. 检查运营侧的产品缺口

若客户痛点来自缺少的运营界面，明确指出。常见示例：

- 无计费门户
- 无使用量/速率限制可见性
- 无计划/席位说明
- 无取消流程
- 无重复订阅防护

将这些视为 ECC 或网站的后续事项，而非仅仅是支持事件。

### 5. 生成运营移交报告

最后提供：

- 客户状态摘要
- 已采取的行动
- 收入影响
- 要发送的后续跟进文字
- 要创建的产品或待办事项

## 输出格式

使用以下结构：

```text
CUSTOMER
- name / email
- relevant account identifiers

BILLING STATE
- active subscriptions
- invoice or renewal state
- anomalies

DECISION
- issue classification
- why this action is correct

ACTION TAKEN
- refund / cancel / portal / no-op

FOLLOW-UP
- short customer message

PRODUCT GAP
- what should be fixed in the product or website
```

## 良好建议示例

- "正确的修复方案是计费门户，而非目前就建自定义仪表盘"
- "这看起来是重复的个人结账，而非真实的团队席位购买"
- "退款一笔重复费用，保留剩余的活跃订阅，稍后若需要再将客户转换为组织计费"
