# Customer Billing Ops

本技能用于真实的客户操作，而非通用的支付 API 设计。

目标是帮助运营人员回答：这位客户是谁，发生了什么，最安全的修复方案是什么，应该发送什么跟进信息？

## 何时使用

- 客户反映计费异常、要求退款或无法取消订阅
- 调查重复订阅、意外扣费、续费失败或流失风险
- 审查套餐结构、活跃订阅、年付与月付转化或团队席位混淆
- 创建或验证计费门户流程
- 审计涉及订阅、发票、退款或支付方式的支持投诉

## 首选工具界面

- 优先使用 Stripe 等连接的计费工具
- 仅将邮件、GitHub 或 issue 跟踪器作为辅助证据
- 当平台已提供所需控件时，优先使用托管计费/客户门户，而非自定义账户管理代码

## 安全限制

- 切勿在响应中暴露密钥、完整卡号或不必要的客户个人信息
- 不要盲目退款；首先对问题进行分类
- 区分以下情况：
  - 意外重复购买
  - 有意的多席位或团队购买
  - 产品有问题 / 未达到预期价值
  - 结账失败或未完成
  - 因缺少自助服务控件而取消
- 对于年付套餐、团队套餐和按比例计算的状态，在采取行动前验证合同形态

## 工作流程

### 1. 准确识别客户

从最强的标识符开始：

- 客户邮箱
- Stripe 客户 ID
- 订阅 ID
- 发票 ID
- 若已知映射回计费，使用 GitHub 用户名或支持邮箱

返回简洁的身份摘要：

- 客户信息
- 活跃订阅
- 已取消订阅
- 发票
- 明显异常，如重复活跃订阅

### 2. 对问题进行分类

在采取行动前，将案例归入一个类别：

| 案例 | 典型操作 |
|------|----------------|
| 重复的个人订阅 | 取消多余订阅，考虑退款 |
| 真实的多席位/团队意图 | 保留席位，说明计费模型 |
| 支付失败 / 结账未完成 | 通过门户恢复或更新支付方式 |
| 缺少自助服务控件 | 提供门户、取消路径或发票访问 |
| 产品故障或信任破裂 | 退款、道歉、记录产品问题 |

### 3. 优先采取最安全的可逆操作

优先顺序：

1. 恢复自助管理功能
2. 修复重复或损坏的计费状态
3. 仅退还受影响的费用或重复扣款
4. 记录原因
5. 发送简短的客户跟进信息

如果修复需要产品工作，分开处理：

- 立即处理客户补救
- 将产品 Bug / 工作流缺口放入待办事项

### 4. 检查运营方的产品缺口

如果客户痛点来自运营方缺失的界面，明确指出。常见示例：

- 没有计费门户
- 没有用量/限额可见性
- 没有套餐/席位说明
- 没有取消流程
- 没有重复订阅防护

将这些视为 ECC 或网站的后续事项，而非单纯的支持事件。

### 5. 生成运营人员交接内容

以以下内容结尾：

- 客户状态摘要
- 已采取的操作
- 收入影响
- 待发送的跟进文本
- 需要创建的产品或待办事项 issue

## 输出格式

使用以下结构：

```text
客户信息
- 姓名 / 邮箱
- 相关账户标识符

计费状态
- 活跃订阅
- 发票或续费状态
- 异常情况

决策
- 问题分类
- 为何此操作正确

已采取的操作
- 退款 / 取消 / 门户 / 不操作

跟进内容
- 简短的客户消息

产品缺口
- 产品或网站中应修复的内容
```

## 优质建议示例

- "正确的修复方案是计费门户，现在还不需要自定义仪表板"
- "这看起来像是重复的个人结账，而非真实的团队席位购买"
- "退还一笔重复扣款，保留其余的活跃订阅，之后如有需要再将客户迁移到组织计费"
