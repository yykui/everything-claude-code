# SEO

通过技术正确性、性能和内容相关性提升搜索可见度，而非依赖技巧。

## 何时使用

在以下情况使用此技能：
- 审计爬取性、索引性、canonical 标签或重定向
- 改进 title 标签、meta 描述和标题结构
- 添加或验证结构化数据
- 改进 Core Web Vitals
- 进行关键词研究并将关键词映射到 URL
- 规划内部链接或 sitemap / robots 变更

## 工作原理

### 原则

1. 在内容优化之前修复技术阻塞问题。
2. 一个页面应有一个明确的主要搜索意图。
3. 优先考虑长期质量信号，而非操纵性模式。
4. 移动端优先假设很重要，因为索引是移动端优先的。
5. 建议应针对具体页面且可实施。

### 技术 SEO 清单

#### 爬取性

- `robots.txt` 应允许重要页面并屏蔽低价值界面
- 重要页面不应被意外设置为 `noindex`
- 重要页面应在浅层点击深度内可达
- 避免超过两跳的重定向链
- canonical 标签应自洽且不形成循环

#### 索引性

- 首选 URL 格式应保持一致
- 多语言页面在使用时需要正确的 hreflang
- sitemap 应反映预期的公开界面
- 重复 URL 不应在没有 canonical 控制的情况下相互竞争

#### 性能

- LCP < 2.5s
- INP < 200ms
- CLS < 0.1
- 常见修复：预加载主图资源、减少阻塞渲染的工作、保留布局空间、精简大型 JS

#### 结构化数据

- 首页：适当时使用 organization 或 business schema
- 编辑页面：`Article` / `BlogPosting`
- 产品页面：`Product` 和 `Offer`
- 内页：`BreadcrumbList`
- 问答部分：仅在内容真正匹配时使用 `FAQPage`

### 页面规则

#### Title 标签

- 目标约 50-60 个字符
- 将主要关键词或概念放在前面
- 标题对人类可读，而不是为机器人堆砌

#### Meta 描述

- 目标约 120-160 个字符
- 诚实描述页面内容
- 自然包含主要话题

#### 标题结构

- 一个清晰的 `H1`
- `H2` 和 `H3` 应反映实际内容层级
- 不要仅为视觉样式而跳过结构

### 关键词映射

1. 定义搜索意图
2. 收集现实的关键词变体
3. 按意图匹配度、潜在价值和竞争程度排优先级
4. 将一个主要关键词/主题映射到一个 URL
5. 检测并避免关键词互食

### 内部链接

- 从强势页面链接到你希望排名的页面
- 使用描述性锚文本
- 在可以使用更具体锚文本时避免通用锚文本
- 从新页面补充链接到相关现有页面

## 示例

### Title 公式

```text
主要话题 - 具体修饰词 | 品牌
```

### Meta 描述公式

```text
行动 + 话题 + 价值主张 + 一个支撑细节
```

### JSON-LD 示例

```json
{
  "@context": "https://schema.org",
  "@type": "Article",
  "headline": "Page Title Here",
  "author": {
    "@type": "Person",
    "name": "Author Name"
  },
  "publisher": {
    "@type": "Organization",
    "name": "Brand Name"
  }
}
```

### 审计输出格式

```text
[HIGH] 产品页面存在重复 title 标签
位置：src/routes/products/[slug].tsx
问题：动态标题折叠为相同的默认字符串，削弱了相关性并产生重复信号。
修复：使用产品名称和主要类别为每个产品生成唯一标题。
```

## 反模式

| 反模式 | 修复方案 |
|--------|---------|
| 关键词堆砌 | 优先为用户写作 |
| 近似重复的薄内容页面 | 合并或差异化 |
| 与实际内容不匹配的 Schema | 使 Schema 与现实对应 |
| 不先检查实际页面就给出内容建议 | 先阅读真实页面 |
| 通用的"改进 SEO"输出 | 将每条建议与具体页面或资产绑定 |
