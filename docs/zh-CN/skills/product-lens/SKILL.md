# 产品视角 — 先思考后构建

此通道负责产品诊断，而非编写实现就绪的规格文档。

如果用户需要持久化的 PRD 到 SRS 或能力契约制品，请移交给 `product-capability`。

## 何时使用

- 开始任何功能之前 — 验证"为什么"
- 每周产品评审 — 我们是否在构建正确的东西？
- 在多个功能之间难以取舍时
- 发布前 — 对用户旅程进行合理性检查
- 将模糊想法转化为工程规划前的产品简报时

## 工作原理

### 模式一：产品诊断

类似自动化的 YC 办公时间。提出关键问题：

```
1. 这是为谁做的？（具体的人，而非"开发者"）
2. 痛点是什么？（量化：多频繁、有多严重、他们现在怎么做？）
3. 为什么是现在？（什么变化让这件事成为可能/必要？）
4. 十星版本是什么？（如果时间和金钱不是问题）
5. MVP 是什么？（能验证假设的最小可行产品）
6. 反目标是什么？（你明确不构建的是什么？）
7. 你怎么知道它奏效了？（指标，而非感觉）
```

输出：包含答案、风险和是否推进建议的 `PRODUCT-BRIEF.md`。

如果结论是"是的，构建它"，下一个通道是 `product-capability`，而不是更多的创始人演讲。

### 模式二：创始人评审

从创始人视角审视你的当前项目：

```
1. 阅读 README、CLAUDE.md、package.json、近期提交
2. 推断：这想成为什么？
3. 评分：产品市场契合信号（0-10）
   - 使用增长轨迹
   - 留存指标（重复贡献者、回访用户）
   - 收入信号（定价页面、计费代码、Stripe 集成）
   - 竞争护城河（什么难以复制？）
4. 识别：能让产品提升十倍的一件事
5. 标记：你正在构建但无关紧要的东西
```

### 模式三：用户旅程审计

映射真实用户体验：

```
1. 以新用户身份克隆/安装产品
2. 记录每个摩擦点（令人困惑的步骤、错误、缺失文档）
3. 计时每个步骤
4. 与竞争对手的引导流程对比
5. 评分：价值实现时间（用户获得第一次成功需要多久？）
6. 建议：引导流程的前三项改进
```

### 模式四：功能优先级排序

当有十个想法需要选择两个时：

```
1. 列出所有候选功能
2. 按以下标准评分：影响力（1-5）× 把握度（1-5）÷ 工作量（1-5）
3. 按 ICE 分数排名
4. 应用约束条件：资金跑道、团队规模、依赖关系
5. 输出：附带理由的优先路线图
```

## 输出

所有模式输出可操作的文档，而非文章。每条建议都有具体的下一步行动。

## 集成

搭配使用：
- `/browser-qa` 验证用户旅程审计发现
- `/design-system audit` 评估视觉打磨程度
- `/canary-watch` 用于发布后监控
- `product-capability` 当产品简报需要成为实现就绪的能力计划时
