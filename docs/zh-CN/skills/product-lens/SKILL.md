# 产品透镜——先思考，再构建

本通道负责产品诊断，而非编写可落地的规范。

如果用户需要持久化的 PRD-to-SRS 或能力契约制品，请交接给 `product-capability`。

## 何时使用

- 启动任何功能之前——验证"为什么"
- 每周产品评审——我们是否在构建正确的东西？
- 在多个功能之间难以抉择时
- 发布前——对用户旅程进行完整性检查
- 将模糊想法转化为产品简报，然后再进入工程规划

## 工作原理

### 模式 1：产品诊断

类似 YC 办公室会议，但自动化。提出关键问题：

```
1. 这是为谁设计的？（具体的人，而非"开发者"）
2. 痛点是什么？（量化：频率、严重程度、他们现在如何应对？）
3. 为什么是现在？（是什么变化让这成为可能或必要？）
4. 10 星版本是什么？（如果不受金钱/时间限制）
5. MVP 是什么？（能证明假设的最小可行方案）
6. 反目标是什么？（明确不构建的内容）
7. 如何判断是否成功？（指标，而非感觉）
```

输出：包含答案、风险和可行/不可行建议的 `PRODUCT-BRIEF.md`。

如果结论是"是的，构建这个"，下一个通道是 `product-capability`，而非更多的创始人剧场。

### 模式 2：创始人视角评审

通过创始人视角评审当前项目：

```
1. 读取 README、CLAUDE.md、package.json、近期提交
2. 推断：这想成为什么？
3. 评分：产品市场契合信号（0-10）
   - 使用增长轨迹
   - 留存指标（重复贡献者、回访用户）
   - 收入信号（定价页面、计费代码、Stripe 集成）
   - 竞争护城河（什么是难以复制的？）
4. 识别：能使其增长 10 倍的单一因素
5. 标记：正在构建但无关紧要的内容
```

### 模式 3：用户旅程审计

映射真实的用户体验：

```
1. 以新用户身份克隆/安装产品
2. 记录每个摩擦点（令人困惑的步骤、错误、缺失文档）
3. 计时每个步骤
4. 与竞争对手的入门体验进行比较
5. 评分：价值实现时间（用户首次获得成果需要多久？）
6. 推荐：入门体验的 3 个最优先改进项
```

### 模式 4：功能优先级排序

当有 10 个想法需要选出 2 个时：

```
1. 列出所有候选功能
2. 对每项评分：影响力（1-5）× 信心（1-5）÷ 工作量（1-5）
3. 按 ICE 分数排序
4. 应用约束条件：资金跑道、团队规模、依赖关系
5. 输出：带有理由的优先级路线图
```

## 输出

所有模式输出的是可执行文档，而非长篇论文。每项建议都有具体的下一步行动。

## 集成

与以下技能配合使用：
- `/browser-qa` 验证用户旅程审计发现
- `/design-system audit` 进行视觉精细度评估
- `/canary-watch` 用于发布后监控
- `product-capability` 当产品简报需要转化为可落地的能力计划时
