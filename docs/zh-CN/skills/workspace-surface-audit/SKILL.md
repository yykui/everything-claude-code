# 工作区界面审计

只读审计技能，用于回答"这个工作区和机器现在实际能做什么，接下来我们应该添加或启用什么？"

这是 ECC 对设置审计插件的原生替代方案。除非用户明确要求后续实施，否则它不修改文件。

## 何时使用

- 用户说"设置 Claude Code"、"推荐自动化"、"我应该使用什么插件或 MCP？"或"我缺少什么？"
- 在安装更多技能、hooks 或连接器之前审计机器或仓库
- 将官方市场插件与 ECC 原生覆盖进行比较
- 审查 `.env`、`.mcp.json`、插件设置或连接应用界面以找到缺失的工作流层
- 决定某个功能应该是技能、hook、Agent、MCP 还是外部连接器

## 不可违反的规则

- 永远不要打印密钥值。只呈现提供商名称、功能名称、文件路径，以及密钥或配置是否存在。
- 当 ECC 能合理地拥有该界面时，优先选择 ECC 原生工作流而非通用的"安装另一个插件"建议。
- 将外部插件视为基准和灵感，而非权威的产品边界。
- 清晰地分离三件事：
  - 现在已可用
  - 可用但在 ECC 中封装不佳
  - 不可用且需要新集成

## 审计输入

仅检查回答问题所需的文件和设置：

1. 仓库界面
   - `package.json`、锁文件、语言标记、框架配置、`README.md`
   - `.mcp.json`、`.lsp.json`、`.claude/settings*.json`、`.codex/*`
   - `AGENTS.md`、`CLAUDE.md`、安装清单、hook 配置
2. 环境界面
   - 活动仓库和明显相邻 ECC 工作区中的 `.env*` 文件
   - 仅呈现密钥名称，如 `STRIPE_API_KEY`、`TWILIO_AUTH_TOKEN`、`FAL_KEY`
3. 连接工具界面
   - 已安装的插件、已启用的连接器、MCP 服务器、LSP 和应用集成
4. ECC 界面
   - 已涵盖该需求的现有技能、命令、hooks、Agent 和安装模块

## 审计流程

### 阶段一：盘点现有内容

生成简洁的清单：

- 活动工具链目标
- 已安装的插件和连接应用
- 已配置的 MCP 服务器
- 已配置的 LSP 服务器
- 密钥名称暗示的环境支持服务
- 与工作区相关的现有 ECC 技能

如果某个界面仅作为原语存在，请指出。示例：

- "Stripe 通过连接应用可用，但 ECC 缺少计费操作员技能"
- "Google Drive 已连接，但没有 ECC 原生的 Google Workspace 操作员工作流"

### 阶段二：对比官方和已安装界面进行基准测试

将工作区与以下内容进行比较：

- 与设置、审查、文档、设计或工作流质量重叠的官方 Claude 插件
- Claude 或 Codex 中本地安装的插件
- 用户当前连接的应用界面

不要只列出名称。对每个比较，回答：

1. 它们实际做什么
2. ECC 是否已有对等功能
3. ECC 是否只有原语
4. ECC 是否完全缺少该工作流

### 阶段三：将差距转化为 ECC 决策

对每个真实差距，推荐正确的 ECC 原生形态：

| 差距类型 | 首选 ECC 形态 |
|----------|--------------|
| 可重复的操作员工作流 | 技能 |
| 自动执行或副作用 | Hook |
| 专门的委托角色 | Agent |
| 外部工具桥接 | MCP 服务器或连接器 |
| 安装/引导指导 | 设置或审计技能 |

当需求是操作性而非基础设施性时，默认使用协调现有工具的面向用户技能。

## 输出格式

按以下顺序返回五个部分：

1. **当前界面**
   - 现在已可用的内容
2. **对等性**
   - ECC 已匹配或超越基准的地方
3. **仅有原语的差距**
   - 工具存在，但 ECC 缺少清晰的操作员技能
4. **缺失集成**
   - 尚不可用的功能
5. **前 3-5 个下一步行动**
   - 具体的 ECC 原生添加，按影响力排序

## 推荐规则

- 每个类别最多推荐 1-2 个最高价值想法。
- 优先推荐具有明显用户意图和商业价值的技能：
  - 设置审计
  - 计费/客户操作
  - 问题/项目操作
  - Google Workspace 操作
  - 部署/运维控制
- 如果连接器是公司特定的，仅在它对用户工作流确实可用或明显有用时推荐。
- 如果 ECC 已有强大的原语，提议一个封装技能而非发明全新的子系统。

## 良好结果

- 用户可以立即看到已连接什么、缺少什么，以及 ECC 下一步应拥有什么。
- 推荐足够具体，无需另一次发现过程即可在仓库中实施。
- 最终答案围绕工作流组织，而非 API 品牌。
