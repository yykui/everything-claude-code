# API 连接器构建器

当任务是添加仓库原生集成界面而非通用 HTTP 客户端时使用此技能。

关键在于匹配宿主仓库的模式：

- 连接器布局
- 配置模式
- 认证模型
- 错误处理
- 测试风格
- 注册/发现连接

## 何时使用

- "为这个项目构建一个 Jira 连接器"
- "按照现有模式添加一个 Slack 提供者"
- "为这个 API 创建新集成"
- "构建一个匹配仓库连接器风格的插件"

## 护栏

- 当仓库已有集成架构时，不要发明新的集成架构
- 不要仅从供应商文档开始；先从仓库内现有连接器开始
- 如果仓库期望注册连接、测试和文档，不要止步于传输代码
- 如果仓库有更新的当前模式，不要盲目复制旧连接器

## 工作流

### 1. 学习代码风格

检查至少 2 个现有连接器/提供者并映射：

- 文件布局
- 抽象边界
- 配置模型
- 重试/分页约定
- 注册钩子
- 测试夹具和命名

### 2. 缩小目标集成范围

只定义仓库实际需要的界面：

- 认证流程
- 关键实体
- 核心读写操作
- 分页和频率限制
- Webhook 或轮询模型

### 3. 按仓库原生层构建

典型切片：

- 配置/模式
- 客户端/传输
- 映射层
- 连接器/提供者入口点
- 注册
- 测试

### 4. 对照源模式验证

新连接器在代码库中应显而易见，而不是从不同生态系统导入的。

## 参考形态

### 提供者风格

```text
providers/
  existing_provider/
    __init__.py
    provider.py
    config.py
```

### 连接器风格

```text
integrations/
  existing/
    client.py
    models.py
    connector.py
```

### TypeScript 插件风格

```text
src/integrations/
  existing/
    index.ts
    client.ts
    types.ts
    test.ts
```
