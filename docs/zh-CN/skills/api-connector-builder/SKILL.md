# API 连接器构建器

当任务是添加仓库原生集成表面，而非通用 HTTP 客户端时，请使用本技能。

重点是匹配宿主仓库的模式：

- 连接器布局
- 配置模式
- 认证模型
- 错误处理
- 测试风格
- 注册/发现接线

## 何时使用

- "为这个项目构建一个 Jira 连接器"
- "按照现有模式添加一个 Slack 提供商"
- "为这个 API 创建一个新集成"
- "构建一个匹配仓库连接器风格的插件"

## 使用规范

- 当仓库已有集成架构时，不要自行发明新的集成架构
- 不要单从供应商文档开始；先从仓库内现有连接器开始
- 如果仓库期望有注册接线、测试和文档，不要止步于传输层代码
- 如果仓库有更新的当前模式，不要盲目复制旧连接器

## 工作流

### 1. 学习代码风格

检查至少 2 个现有连接器/提供商并映射：

- 文件布局
- 抽象边界
- 配置模型
- 重试/分页约定
- 注册钩子
- 测试夹具和命名规范

### 2. 确定目标集成范围

仅定义仓库实际需要的表面：

- 认证流程
- 核心实体
- 核心读/写操作
- 分页和频率限制
- Webhook 或轮询模型

### 3. 按仓库原生层构建

典型切片：

- 配置/模式
- 客户端/传输层
- 映射层
- 连接器/提供商入口点
- 注册
- 测试

### 4. 对照源模式验证

新连接器在代码库中看起来应该是显而易见的，而不是从另一个生态系统移植过来的。

## 参考结构

### 提供商风格

```text
providers/
  existing_provider/
    __init__.py
    provider.py
    config.py
```

### 连接器风格

```text
integrations/
  existing/
    client.py
    models.py
    connector.py
```

### TypeScript 插件风格

```text
src/integrations/
  existing/
    index.ts
    client.ts
    types.ts
    test.ts
```
