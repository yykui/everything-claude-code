# 统一通知操作

当真正的问题不是缺少一个通知时，使用本技能。真正的问题是碎片化的通知系统。

工作目标是将分散的事件转化为一个具有以下特征的操作员界面：
- 清晰的严重级别
- 清晰的所有权
- 清晰的路由
- 清晰的后续行动

## 何时使用

- 用户希望在 GitHub、Linear、本地 hooks、桌面提醒、聊天或邮件之间建立统一通知通道
- CI 失败、审查请求、问题更新和操作员事件到达各自分离的地方
- 当前设置产生噪音而非行动
- 用户希望将重叠的通知分支或积压提案合并到一个 ECC 原生通道
- 工作区已有 hooks、MCP 或连接工具，但没有一致的通知策略

## 优先使用的界面

从已存在的内容开始：
- GitHub 问题、PR、审查、评论和 CI
- Linear 问题/项目流转
- 本地 hook 事件和会话生命周期信号
- 桌面通知原语
- 实际存在时的连接邮件/聊天界面

优先使用 ECC 原生编排，而非建议用户采用单独的通知产品。

## 不可违反的规则

- 永远不要暴露 token、密钥、webhook 密钥或内部标识符
- 分离：
  - 事件来源
  - 严重级别
  - 路由通道
  - 操作员行动
- 当中断成本不明确时默认为摘要优先
- 不要将每个事件扇出到每个通道
- 如果真正的修复是更好的问题分类、hook 策略或项目流程，请明确说明

## 事件管道

将通道视为：

1. **捕获**事件
2. **分类**紧迫性和所有者
3. **路由**到正确的通道
4. **合并**重复内容和低信号噪音
5. **附加**下一个操作员行动

目标是更少、更好的通知。

## 默认严重级别模型

| 级别 | 示例 | 默认处理 |
| --- | --- | --- |
| 严重 | 默认分支 CI 中断、安全问题、发布阻塞、部署失败 | 立即中断 |
| 高 | 请求审查、PR 失败、所有者阻塞交接 | 当日提醒 |
| 中 | 问题状态变更、重要评论、积压流转 | 摘要或队列 |
| 低 | 重复成功、常规噪音、冗余生命周期标记 | 抑制或折叠 |

如果工作区没有严重级别模型，在提出自动化建议之前先建立一个。

## 工作流程

### 1. 盘点当前界面

列出：
- 事件来源
- 当前通道
- 发出告警的现有 hooks/脚本
- 同一事件的重复路径
- 重要内容未被呈现的静默失败情况

标注 ECC 已拥有的内容。

### 2. 决定什么值得中断

对每个事件系列回答：
- 谁需要知道？
- 他们需要多快知道？
- 这应该中断、批处理还是仅记录？

使用以下默认值：
- 对发布、CI、安全和所有者阻塞事件进行中断
- 对中等信号更新进行摘要
- 对遥测和低信号生命周期标记仅记录

### 3. 在添加通道之前先合并重复内容

寻找：
- 同一 PR 事件出现在 GitHub、Linear 和本地日志中
- 同一失败的重复 hook 通知
- 应该汇总而非原始转发的评论或状态噪音
- 互相重复而不提供更好行动路径的通道

优先：
- 一个规范摘要
- 一个所有者
- 一个主要通道
- 一个备用路径

### 4. 设计 ECC 原生工作流

对每个真实通知需求，定义：
- **来源**
- **门控条件**
- **形态**：即时告警、摘要、队列或仅仪表板
- **通道**
- **行动**

如果 ECC 已有该原语，优先使用：
- 操作员分类技能
- 自动发射/执行的 hook
- 委托分类的 Agent
- 仅在真正缺少桥接时使用 MCP/连接器

### 5. 返回以行动为导向的设计

以以下内容结束：
- 保留什么
- 抑制什么
- 合并什么
- ECC 下一步应封装什么

## 输出格式

```text
CURRENT SURFACE
- sources
- channels
- duplicates
- gaps

EVENT MODEL
- critical
- high
- medium
- low

ROUTING PLAN
- source -> channel
- why
- operator owner

CONSOLIDATION
- suppress
- merge
- canonical summaries

NEXT ECC MOVE
- skill / hook / agent / MCP
- exact workflow to build next
```

## 推荐规则

- 优先选择一个强通道而非多个弱通道
- 对中低信号更新优先使用摘要
- 当信号应自动发射时优先使用 hooks
- 当工作是分类、路由和审查优先决策时优先使用操作员技能
- 当根本原因是积压/PR 协调而非告警时优先使用 `project-flow-ops`
- 当用户首先需要来源清单时优先使用 `workspace-surface-audit`
- 如果桌面通知已足够，不要发明不必要的外部桥接

## 良好使用场景

- "我们有 GitHub、Linear 和本地 hook 告警，但没有单一的操作员流程"
- "我们的 CI 失败很嘈杂，人们忽视它们"
- "我想要一个跨 Claude、OpenCode 和 Codex 界面的通知策略"
- "确定什么应该中断，什么应该进入摘要"
- "将重叠的通知 PR 想法合并成一个规范的 ECC 通道"
