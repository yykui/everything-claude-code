# Terminal Ops

当用户需要真实的仓库执行时使用本技能：运行命令、检查 git 状态、调试 CI 或构建问题、进行小范围修复，并精确报告变更内容和验证结果。

本技能有意比通用编码指导更窄。它是以证据为先的终端执行操作员工作流。

## 技能栈

在相关时，将以下 ECC 原生技能纳入工作流：

- `verification-loop`：用于变更后的精确验证步骤
- `tdd-workflow`：当正确修复需要回归覆盖时
- `security-review`：涉及密钥、认证或外部输入时
- `github-ops`：任务依赖 CI 运行、PR 状态或发布状态时
- `knowledge-ops`：需要将已验证结果保存到持久项目上下文时

## 何时使用

- 用户说"修复"、"调试"、"运行这个"、"检查仓库"或"推送"
- 任务依赖命令输出、git 状态、测试结果或已验证的本地修复
- 答案必须区分：已本地变更、已本地验证、已提交和已推送

## 操作守则

- 编辑前先检查
- 用户仅请求审计/审查时保持只读
- 优先使用仓库本地脚本和辅助工具，而非临时包装器
- 在重新运行验证命令之前不声称已修复
- 在分支实际移动到上游之前不声称已推送

## 工作流程

### 1. 确定工作面

明确：

- 确切的仓库路径
- 分支
- 本地 diff 状态
- 请求模式：
  - 检查
  - 修复
  - 验证
  - 推送

### 2. 先读取失败面

在做任何变更前：

- 检查错误
- 检查文件或测试
- 检查 git 状态
- 在盲目重新读取之前使用已提供的日志或上下文

### 3. 保持修复范围小

每次解决一个主要失败：

- 首先使用最小的有效验证命令
- 仅在解决本地失败后才升级到更大的构建/测试流程
- 如果命令持续以相同的模式失败，停止大范围重试并缩小范围

### 4. 报告精确执行状态

使用精确的状态词：

- 已检查
- 已本地变更
- 已本地验证
- 已提交
- 已推送
- 已阻塞

## 输出格式

```text
SURFACE
- repo
- branch
- requested mode

EVIDENCE
- failing command / diff / test

ACTION
- what changed

STATUS
- inspected / changed locally / verified locally / committed / pushed / blocked
```

## 注意事项

- 当可以读取实时仓库状态时，不要依赖过时的记忆
- 不要将小范围修复扩大为仓库范围的大改动
- 不要使用破坏性的 git 命令
- 不要忽略不相关的本地工作

## 验证

- 响应中应命名验证命令或测试
- git 相关工作应命名仓库路径和分支
- 任何推送声明应包含目标分支和精确结果
