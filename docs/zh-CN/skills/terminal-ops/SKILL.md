# Terminal Ops

当用户需要真实的仓库执行时使用本技能：运行命令、检查 git 状态、调试 CI 或构建失败、进行窄范围修复，并精确报告发生了哪些变更及验证了哪些内容。

本技能有意比通用编码指导更窄。它是一个面向证据优先终端执行的操作者工作流。

## 技能栈

在工作流中遇到相关情况时，引入以下 ECC 原生技能：

- `verification-loop`：用于变更后的精确验证步骤
- `tdd-workflow`：当正确修复需要回归覆盖时
- `security-review`：当涉及密钥、认证或外部输入时
- `github-ops`：当任务依赖 CI 运行、PR 状态或发布状态时
- `knowledge-ops`：当需要将已验证的结果捕获到持久项目上下文中时

## 使用时机

- 用户说"修复"、"调试"、"运行这个"、"检查仓库"或"推送它"
- 任务依赖命令输出、git 状态、测试结果或已验证的本地修复
- 答案必须区分：本地已变更、本地已验证、已提交、已推送

## 约束规则

- 先检查再编辑
- 若用户仅要求审计/审查，保持只读模式
- 优先使用仓库本地脚本和辅助工具，而非临时拼凑的包装器
- 在重新运行验证命令之前，不得声称已修复
- 分支未实际移动到上游之前，不得声称已推送

## 工作流程

### 1. 明确工作表面

确定：

- 精确的仓库路径
- 分支
- 本地差异状态
- 请求模式：
  - 检查
  - 修复
  - 验证
  - 推送

### 2. 先读取故障表面

在做任何变更之前：

- 检查错误
- 检查文件或测试
- 检查 git 状态
- 先使用已提供的日志或上下文，再重新盲读

### 3. 保持修复范围窄小

每次只解决一个主要故障：

- 先使用最小有效的验证命令
- 只有在解决本地故障后，才升级到更大范围的构建/测试
- 如果命令持续以相同特征失败，停止大范围重试并缩小范围

### 4. 报告精确的执行状态

使用精确的状态词：

- 已检查（inspected）
- 本地已变更（changed locally）
- 本地已验证（verified locally）
- 已提交（committed）
- 已推送（pushed）
- 已阻塞（blocked）

## 输出格式

```text
SURFACE
- repo
- branch
- requested mode

EVIDENCE
- failing command / diff / test

ACTION
- what changed

STATUS
- inspected / changed locally / verified locally / committed / pushed / blocked
```

## 常见陷阱

- 当可以读取实时仓库状态时，不要依赖过时记忆
- 不要将窄范围修复扩展为全仓库大改动
- 不要使用破坏性 git 命令
- 不要忽略不相关的本地工作

## 验证

- 响应中注明验证命令或测试
- 涉及 git 的工作注明仓库路径和分支
- 任何推送声明包含目标分支和精确结果
