# 消息操作

当任务涉及实时消息检索时使用：iMessage、私信、最近的一次性验证码，或在跟进前检查对话线程。

这不是邮件工作。如果主要处理的是邮箱，请使用 `email-ops`。

## 技能栈

在相关时将以下 ECC 原生技能引入工作流：

- `email-ops`：当消息任务实际上是邮箱工作时
- `connections-optimizer`：当私信线程属于对外网络工作时
- `lead-intelligence`：当实时对话线程应为定向或热路径外联提供信息时
- `knowledge-ops`：当对话线程内容需要被捕获到持久上下文中时

## 何时使用

- 用户说"读我的消息"、"查看短信"、"看看私信"或"找到验证码"
- 任务依赖于实时对话线程或发送到本地消息平台的近期验证码
- 用户需要证明检查了哪个来源或线程

## 安全规范

- 首先确定来源：
  - 本地消息
  - X / 社交私信
  - 其他需要浏览器访问的消息平台
- 不要在未命名来源的情况下声称已检查某个线程
- 如果存在已检查的辅助程序或标准路径，不要临时进行原始数据库访问
- 如果认证或多因素验证阻止了访问，报告确切的阻断原因

## 工作流

### 1. 确定准确的线程

在做任何事之前，先明确：

- 消息平台
- 发件人 / 收件人 / 服务
- 时间窗口
- 任务是检索、检查，还是准备回复

### 2. 起草前先阅读

如果任务可能演变为对外跟进：

- 阅读最新的收件内容
- 识别未解决的问题
- 然后在需要时移交给正确的对外技能

### 3. 将验证码处理为专注检索任务

对于一次性验证码：

- 优先搜索近期本地消息窗口
- 尽可能按服务或发件人缩小范围
- 一旦找到验证码或专注搜索已穷尽即停止

### 4. 报告确切证据

返回：

- 使用的来源
- 对话线程或发件人（如可能）
- 时间窗口
- 确切状态：
  - 已阅读
  - 验证码已找到
  - 被阻断
  - 等待起草回复

## 输出格式

```text
来源
- 消息平台
- 发件人 / 线程 / 服务

结果
- 消息摘要或验证码
- 时间窗口

状态
- 已阅读 / 验证码已找到 / 被阻断 / 等待起草回复
```

## 注意事项

- 不要混淆邮箱工作和私信/短信工作
- 不要在未命名来源的情况下声称已完成检索
- 在请求是查找最近验证码时，不要花时间进行大范围搜索
- 不要在未报告阻断原因的情况下反复重试已被阻断的认证路径

## 验证

- 响应中命名了消息来源
- 响应包含发件人、服务、线程或明确的阻断原因
- 最终状态是明确且有界限的
