# 消息操作

当任务是实时消息检索时使用本技能：iMessage、私信、近期一次性验证码，或在跟进前查看会话线程。

本技能不处理邮件工作。如果主要操作面是邮箱，请使用 `email-ops`。

## 技能栈

在相关时将以下 ECC 原生技能纳入工作流：

- `email-ops`：当消息任务实际上是邮箱工作时
- `connections-optimizer`：当私信线程属于对外网络拓展工作时
- `lead-intelligence`：当实时线程应为定向或温暖路径外联提供依据时
- `knowledge-ops`：当线程内容需要存入持久上下文时

## 何时使用

- 用户说"读取我的消息"、"查看短信"、"看看私信"或"找到验证码"
- 任务依赖于实时线程或发送到本地消息界面的近期验证码
- 用户希望留存具体检查了哪个来源或线程的证据

## 安全准则

- 先确定来源：
  - 本地消息
  - X / 社交私信
  - 其他需要浏览器访问的消息界面
- 未明确说明来源，不声称已检查某线程
- 如果存在已检查的辅助工具或标准路径，不擅自进行原始数据库访问
- 如果认证或多因素验证阻塞了该界面，报告具体阻塞原因

## 工作流

### 1. 确定具体线程

在执行任何操作之前，先明确：

- 消息界面
- 发送方 / 接收方 / 服务
- 时间窗口
- 任务是检索、查看还是准备回复

### 2. 先读后写

如果任务可能转变为对外跟进：

- 先读取最新的入站消息
- 识别未关闭的待处理事项
- 然后根据需要移交给正确的对外技能

### 3. 将验证码处理为专项检索任务

对于一次性验证码：

- 首先在近期本地消息窗口中搜索
- 尽可能按服务或发送方缩小范围
- 找到验证码或专项搜索穷举后即停止

### 4. 报告精确证据

返回：

- 使用的来源
- 线程或发送方（尽可能提供）
- 时间窗口
- 明确状态：
  - 已读取
  - 找到验证码
  - 已阻塞
  - 等待回复草稿

## 输出格式

```text
来源
- 消息界面
- 发送方 / 线程 / 服务

结果
- 消息摘要或验证码
- 时间窗口

状态
- 已读取 / 找到验证码 / 已阻塞 / 等待回复草稿
```

## 注意事项

- 不要混淆邮箱工作和私信/短信工作
- 不要在未说明来源的情况下声称已完成检索
- 任务是查找近期验证码时，不要花时间进行大范围搜索
- 不要在未向用户暴露阻塞问题的情况下反复重试被阻塞的认证路径

## 验证

- 响应中说明了消息来源
- 响应包含发送方、服务、线程或明确的阻塞原因
- 最终状态明确且有边界
