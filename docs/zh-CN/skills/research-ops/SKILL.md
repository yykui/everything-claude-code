# 研究运营

当用户要求研究当前信息、比较选项、丰富人员或公司信息，或将重复性查询转化为监控工作流时，使用本技能。

这是仓库研究技术栈的操作包装器，不是 `deep-research`、`exa-search` 或 `market-research` 的替代品；它告诉你何时以及如何组合使用它们。

## 技能栈

在工作流相关时，引入以下 ECC 原生技能：

- `exa-search` 用于快速当前网络发现
- `deep-research` 用于多源综合与引用
- `market-research` 当最终结果应是建议或排名决策时
- `lead-intelligence` 当任务是人员/公司定向而非通用研究时
- `knowledge-ops` 当结果应事后存储到持久上下文时

## 何时使用

- 用户说"研究"、"查找"、"比较"、"我应该联系谁"或"最新动态"
- 答案取决于当前公开信息
- 用户已提供证据并希望将其纳入最新建议
- 任务可能足够重复，应成为监控而非一次性查询

## 防护措施

- 当最新搜索成本低廉时，不要用过期记忆回答当前问题
- 明确区分：
  - 有来源的事实
  - 用户提供的证据
  - 推断
  - 建议
- 如果答案已在本地代码或文档中，不要启动重量级研究流程

## 工作流

### 1. 从用户已提供的内容开始

将所提供的材料规范化为：

- 已有证据的事实
- 需要验证
- 开放问题

如果用户已构建了部分模型，不要从零重新分析。

### 2. 对需求进行分类

搜索前选择正确的通道：

- 快速事实性回答
- 比较或决策备忘录
- 线索/扩充传递
- 重复监控候选

### 3. 先采用最轻量有效的证据路径

- 使用 `exa-search` 进行快速发现
- 当需要综合或多源时升级到 `deep-research`
- 当结果应以建议结束时使用 `market-research`
- 当真正的需求是目标排名或暖路径发现时交接给 `lead-intelligence`

### 4. 以明确的证据边界报告

对于重要主张，说明它们是：

- 有来源的事实
- 用户提供的上下文
- 推断
- 建议

对时效性敏感的答案应包含具体日期。

### 5. 决定任务是否应保持手动

如果用户很可能反复提出相同的研究问题，明确说明，并建议监控或工作流层，而非永远重复相同的手动搜索。

## 输出格式

```text
QUESTION TYPE
- factual / comparison / enrichment / monitoring

EVIDENCE
- sourced facts
- user-provided context

INFERENCE
- what follows from the evidence

RECOMMENDATION
- answer or next move
- whether this should become a monitor
```

## 常见错误

- 不要在未标注的情况下将推断混入有来源的事实
- 不要忽略用户提供的证据
- 不要对本地仓库上下文能回答的问题使用重量级研究通道
- 不要在没有日期的情况下给出时效性敏感的答案

## 验证

- 重要主张按证据类型标注
- 时效性敏感的输出包含日期
- 最终建议与实际使用的研究模式相匹配
