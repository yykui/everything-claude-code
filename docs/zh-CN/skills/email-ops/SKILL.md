# 邮件运营

当实际任务是邮箱操作时使用本技能：分类整理、草稿编写、回复、发送或确认邮件已发送。

这不是通用写作技能，而是围绕真实邮件界面的操作工作流。

## 技能栈

根据需要将以下 ECC 原生技能引入工作流：

- `brand-voice`：在起草任何面向用户的内容前使用
- `investor-outreach`：用于面向投资者、合作伙伴或赞助商的邮件
- `customer-billing-ops`：当邮件线程涉及计费/支持事件而非普通往来邮件时使用
- `knowledge-ops`：当邮件或线程事后需要纳入持久上下文时使用
- `research-ops`：当回复依赖最新外部信息时使用

## 何时使用

- 用户要求整理收件箱或归档低价值邮件
- 用户需要草拟、回复或新建一封外发邮件
- 用户想知道某封邮件是否已经发送
- 用户需要证明使用了哪个账户、线程或已发送记录

## 操作限制

- 除非用户明确要求实时发送，否则先起草草稿
- 未经真实已发送文件夹或客户端确认，不得声称邮件已发送
- 不要随意切换发件人账户；选择与项目和收件人匹配的账户
- 清理时不要删除不确定的商业邮件
- 如果任务实际上是私信或 iMessage 工作，移交给 `messages-ops`

## 工作流

### 1. 确定具体操作界面

行动前，明确以下内容：

- 使用哪个邮箱账户
- 哪个线程或收件人
- 任务是分类整理、起草、回复还是发送
- 用户要求仅起草还是实时发送

### 2. 编写前先阅读线程

如果是回复：

- 阅读现有线程
- 识别最后一次外发联系
- 识别任何承诺、截止日期或未回答的问题

如果是新建外发邮件：

- 判断熟悉程度
- 选择正确的渠道和发件人账户
- 起草前调用 `brand-voice`

### 3. 起草，然后验证

仅起草时：

- 生成最终文案
- 说明发件人、收件人、主题和目的

实时发送时：

- 先核实确切的最终正文
- 通过所选邮件界面发送
- 确认邮件已进入已发送或等效的已发送副本存储

### 4. 报告精确状态

使用精确的状态词：

- 已起草
- 待审批
- 已发送
- 受阻
- 等待验证

如果发送界面受阻，保留草稿并报告确切的阻塞原因，而非在未说明的情况下改用其他传输方式。

## 输出格式

```text
邮件界面
- 账户
- 线程/收件人
- 请求操作

草稿
- 主题
- 正文

状态
- 已起草 / 已发送 / 受阻
- 适用时提供已发送证明

下一步
- 发送
- 跟进
- 归档/移动
```

## 注意事项

- 未经已发送副本检查，不得声称发送成功
- 不要忽略线程历史而写出缺乏上下文的回复
- 不要将邮箱工作与私信或短信工作流混淆
- 不要暴露密钥、身份验证详情或不必要的邮件元数据

## 验证标准

- 回复中注明账户和线程或收件人
- 任何发送声明包含已发送证明或明确的客户端确认
- 最终状态为以下之一：已起草 / 已发送 / 受阻 / 等待验证
