# 邮件运营

当实际任务是邮箱操作时使用：分流、起草、回复、发送，或确认邮件已进入已发送文件夹。

这不是通用写作技能，而是围绕实际邮件界面的运营工作流。

## 技能栈

在工作流中按需引入以下 ECC 原生技能：

- `brand-voice`：在起草任何面向用户的内容之前
- `investor-outreach`：用于面向投资者、合作伙伴或赞助商的邮件
- `customer-billing-ops`：当邮件线程是计费/支持事件而非普通往来时
- `knowledge-ops`：当邮件或线程需要在事后存入持久上下文时
- `research-ops`：当回复依赖最新的外部事实时

## 何时使用

- 用户要求整理收件箱或归档低价值邮件
- 用户需要起草、回复或发送新的外发邮件
- 用户想确认某封邮件是否已发送
- 用户需要明确使用了哪个账户、哪个线程或哪条已发送记录

## 操作规范

- 除非用户明确要求立即发送，否则先起草
- 未经真实已发送文件夹或客户端确认，不得声称邮件已发送
- 不得随意切换发件账户；选择与项目和收件人匹配的账户
- 清理过程中不要删除不确定的业务邮件
- 若任务实际上是 DM 或 iMessage，转交给 `messages-ops`

## 工作流

### 1. 确认精确的操作界面

行动前，先确定：

- 哪个邮箱账户
- 哪个线程或收件人
- 任务是分流、起草、回复还是发送
- 用户希望仅起草还是实际发送

### 2. 起草前先阅读线程

若是回复：

- 阅读现有线程
- 确认上次对外联系的时间
- 识别承诺、截止日期或未答问题

若是发起新的外发邮件：

- 判断关系亲疏程度
- 选择正确的渠道和发件账户
- 起草前引入 `brand-voice`

### 3. 起草后验证

仅起草时：

- 生成最终文案
- 说明发件人、收件人、主题和目的

实际发送时：

- 先核实最终正文
- 通过选定的邮件界面发送
- 确认邮件已进入已发送文件夹或等效的发送副本存储

### 4. 报告精确状态

使用明确的状态词：

- 已起草
- 待审批
- 已发送
- 已阻断
- 等待验证

若发送界面被阻断，保留草稿并报告具体阻断原因，不要擅自切换备用传输方式而不加说明。

## 输出格式

```text
MAIL SURFACE
- account
- thread / recipient
- requested action

DRAFT
- subject
- body

STATUS
- drafted / sent / blocked
- proof of Sent when applicable

NEXT STEP
- send
- follow up
- archive / move
```

## 注意事项

- 不要在未检查已发送副本的情况下声称发送成功
- 不要忽略线程历史而写出缺乏上下文的回复
- 不要将邮箱工作与 DM 或短信工作流混淆
- 不要暴露密钥、身份验证详情或不必要的邮件元数据

## 验证

- 响应中需注明账户以及线程或收件人
- 任何发送声明都须附带已发送证明或明确的客户端确认
- 最终状态须为以下之一：已起草 / 已发送 / 已阻断 / 等待验证
