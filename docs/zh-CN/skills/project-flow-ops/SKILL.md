# 项目流程运营

此技能将分散的 GitHub issue、PR 和 Linear 任务整合为统一的执行流。

当问题是协调而非编码时使用此技能。

## 何时使用

- 对开放的 PR 或 issue 待办事项进行分类
- 决定哪些内容属于 Linear，哪些应保留在 GitHub
- 将活跃的 GitHub 工作关联到内部执行通道
- 将 PR 分类为合并、移植/重建、关闭或搁置
- 审计评审意见、CI 失败或过时 issue 是否阻塞执行

## 运营模型

- **GitHub** 是公开和社区的事实来源
- **Linear** 是活跃计划工作的内部执行事实来源
- 并非每个 GitHub issue 都需要对应 Linear issue
- 仅在以下情况创建或更新 Linear：
  - 工作处于活跃状态
  - 已委派执行
  - 已安排计划
  - 跨职能协作
  - 重要到需要内部追踪

## 核心工作流

### 1. 首先读取公开界面

收集：

- GitHub issue 或 PR 状态
- 作者和分支状态
- 评审意见
- CI 状态
- 关联 issue

### 2. 对工作进行分类

每个条目应归入以下状态之一：

| 状态 | 含义 |
|------|------|
| 合并 | 自包含、符合策略、已就绪 |
| 移植/重建 | 有价值的想法，但应在 ECC 内手动重新落地 |
| 关闭 | 方向错误、过时、不安全或重复 |
| 搁置 | 可能有价值，但当前未安排 |

### 3. 决定是否需要 Linear

仅在以下情况创建或更新 Linear：

- 执行已在积极规划中
- 涉及多个代码库或工作流
- 工作需要内部负责人或排期
- issue 是更大项目通道的一部分

不要机械地全量同步。

### 4. 保持两个系统一致

当工作处于活跃状态时：

- GitHub issue/PR 应公开说明正在发生什么
- Linear 应在内部追踪负责人、优先级和执行通道

当工作完成或被拒绝时：

- 将公开决议反馈到 GitHub
- 相应标记 Linear 任务

## 评审规则

- 不得仅凭标题、摘要或信任合并；使用完整差异
- 外部来源的功能如果有价值但不是自包含的，应在 ECC 内重建
- CI 红色意味着分类并修复或阻塞；不得假装已就绪合并
- 如果真正的阻塞是产品方向问题，直接说明，而不是躲在工具问题后面

## 输出格式

返回：

```text
PUBLIC STATUS
- issue / PR 状态
- CI / 评审状态

CLASSIFICATION
- 合并 / 移植重建 / 关闭 / 搁置
- 一段话理由

LINEAR ACTION
- 创建 / 更新 / 不需要 Linear 条目
- 适用的项目 / 通道

NEXT OPERATOR ACTION
- 确切的下一步行动
```

## 典型使用场景

- "审计开放的 PR 待办事项，告诉我哪些合并，哪些重建"
- "将 GitHub issue 映射到我们的 ECC 1.x 和 ECC 2.0 项目通道"
- "检查这个是否需要 Linear issue 还是保留在 GitHub"
