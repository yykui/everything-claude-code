# 仪表盘构建器

当任务是构建供人员实际操作的仪表盘时使用此技能。

目标不是"展示所有指标"。目标是回答：

- 系统是否健康？
- 瓶颈在哪里？
- 发生了什么变化？
- 应该采取什么行动？

## 何时使用

- "构建一个 Kafka 监控仪表盘"
- "为 Elasticsearch 创建一个 Grafana 仪表盘"
- "为这个服务制作一个 SigNoz 仪表盘"
- "将这份指标列表转化为真正可操作的仪表盘"

## 安全护栏

- 不要从视觉布局开始；从运营问题开始
- 不要仅因指标存在就把所有可用指标都包含进来
- 不要在没有结构的情况下混合健康、吞吐量和资源面板
- 不要发布没有标题、单位和合理阈值的面板

## 工作流

### 1. 定义运营问题

围绕以下方面组织：

- 健康状况 / 可用性
- 延迟 / 性能
- 吞吐量 / 流量
- 饱和度 / 资源
- 服务特定风险

### 2. 研究目标平台的 Schema

首先检查现有仪表盘：

- JSON 结构
- 查询语言
- 变量
- 阈值样式
- 分区布局

### 3. 构建最小可用看板

推荐结构：

1. 概览
2. 性能
3. 资源
4. 服务特定分区

### 4. 删除虚荣面板

每个面板都应回答一个真实问题。如果不能，则删除它。

## 示例面板集

### Elasticsearch

- 集群健康状态
- 分片分配
- 搜索延迟
- 索引速率
- JVM 堆 / GC

### Kafka

- Broker 数量
- 副本不足的分区
- 消息流入 / 流出
- 消费者延迟
- 磁盘和网络压力

### API 网关 / 入口

- 请求速率
- p50 / p95 / p99 延迟
- 错误率
- 上游健康状态
- 活跃连接数

## 质量检查清单

- [ ] 有效的仪表盘 JSON
- [ ] 清晰的分区分组
- [ ] 标题和单位齐全
- [ ] 阈值/状态颜色有意义
- [ ] 存在常用过滤器的变量
- [ ] 默认时间范围和刷新频率合理
- [ ] 无对运营没有价值的虚荣面板
