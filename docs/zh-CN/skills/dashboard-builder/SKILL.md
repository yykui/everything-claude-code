# Dashboard Builder

当任务是构建一个人们可以用来运营的仪表板时使用本技能。

目标不是"展示所有指标"，而是回答：

- 系统健康吗？
- 瓶颈在哪里？
- 发生了什么变化？
- 应该采取什么行动？

## 何时使用

- "构建一个 Kafka 监控仪表板"
- "为 Elasticsearch 创建一个 Grafana 仪表板"
- "为这个服务制作一个 SigNoz 仪表板"
- "将这个指标列表转化为真正的运营仪表板"

## 安全限制

- 不要从视觉布局开始；从运营问题开始
- 不要仅仅因为指标存在就把所有可用指标都包括进来
- 不要在没有结构的情况下混合健康、吞吐量和资源面板
- 不要发布没有标题、单位和合理阈值的面板

## 工作流程

### 1. 定义运营问题

围绕以下方面组织：

- 健康 / 可用性
- 延迟 / 性能
- 吞吐量 / 流量
- 饱和度 / 资源
- 特定服务的风险

### 2. 研究目标平台架构

首先检查现有仪表板：

- JSON 结构
- 查询语言
- 变量
- 阈值样式
- 区块布局

### 3. 构建最小可用仪表板

推荐结构：

1. 概览
2. 性能
3. 资源
4. 特定服务区块

### 4. 裁剪虚荣面板

每个面板都应该回答一个真实问题。如果不能，就删除它。

## 面板集示例

### Elasticsearch

- 集群健康
- 分片分配
- 搜索延迟
- 索引速率
- JVM 堆 / GC

### Kafka

- Broker 数量
- 欠复制分区
- 消息输入/输出速率
- 消费者滞后
- 磁盘和网络压力

### API 网关 / Ingress

- 请求速率
- p50 / p95 / p99 延迟
- 错误率
- 上游健康状态
- 活跃连接数

## 质量检查清单

- [ ] 有效的仪表板 JSON
- [ ] 清晰的区块分组
- [ ] 标题和单位齐全
- [ ] 阈值/状态颜色有实际意义
- [ ] 常用过滤器有变量设置
- [ ] 默认时间范围和刷新频率合理
- [ ] 没有对运营无价值的虚荣面板
