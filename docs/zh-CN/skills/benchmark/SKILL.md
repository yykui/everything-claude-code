# Benchmark — 性能基线与回归检测

## 何时使用

- 在 PR 前后测量性能影响
- 为项目建立性能基线
- 当用户反映"感觉很慢"时
- 上线前确保达到性能目标
- 将当前技术栈与替代方案进行比较

## 工作原理

### 模式一：页面性能

通过浏览器 MCP 测量真实浏览器指标：

```
1. 导航到每个目标 URL
2. 测量核心 Web 指标（Core Web Vitals）：
   - LCP（最大内容绘制）— 目标 < 2.5s
   - CLS（累积布局偏移）— 目标 < 0.1
   - INP（交互到下次绘制）— 目标 < 200ms
   - FCP（首次内容绘制）— 目标 < 1.8s
   - TTFB（首字节时间）— 目标 < 800ms
3. 测量资源大小：
   - 页面总大小（目标 < 1MB）
   - JS 包大小（目标 < 200KB gzip 后）
   - CSS 大小
   - 图片大小
   - 第三方脚本大小
4. 统计网络请求数
5. 检查阻塞渲染的资源
```

### 模式二：API 性能

对 API 端点进行基准测试：

```
1. 每个端点请求 100 次
2. 测量：p50、p95、p99 延迟
3. 追踪：响应大小、状态码
4. 负载测试：10 个并发请求
5. 与 SLA 目标进行比较
```

### 模式三：构建性能

测量开发反馈循环：

```
1. 冷构建时间
2. 热重载时间（HMR）
3. 测试套件执行时长
4. TypeScript 检查时间
5. 代码检查时间
6. Docker 构建时间
```

### 模式四：前后对比

在变更前后运行以测量影响：

```
/benchmark baseline    # 保存当前指标
# ... 进行变更 ...
/benchmark compare     # 与基线对比
```

输出：
```
| 指标 | 变更前 | 变更后 | 差值 | 结论 |
|--------|--------|-------|-------|---------|
| LCP | 1.2s | 1.4s | +200ms | WARNING: 警告 |
| 包大小 | 180KB | 175KB | -5KB | ✓ 更好 |
| 构建 | 12s | 14s | +2s | WARNING: 警告 |
```

## 输出

将基线以 JSON 格式存储在 `.ecc/benchmarks/` 中。通过 Git 追踪，团队共享基线数据。

## 集成

- CI：在每个 PR 上运行 `/benchmark compare`
- 与 `/canary-watch` 配合进行部署后监控
- 与 `/browser-qa` 配合完成完整的发布前检查清单
