# Code Tour

为代码库导览创建 **CodeTour** `.tour` 文件，这些文件可直接跳转到真实文件和行号范围。导览存放在 `.tours/` 目录中，遵循 CodeTour 格式，而非随意编写的 Markdown 笔记。

一个好的导览是为特定读者设计的叙事：
- 他们正在看什么
- 为什么重要
- 接下来应该走哪条路

只创建 `.tour` JSON 文件。本技能不修改源代码。

## 何时使用

在以下情况使用本技能：
- 用户请求代码导览、入职导览、架构讲解或 PR 导览
- 用户说"解释 X 是如何工作的"并希望获得可复用的引导性产物
- 用户希望为新工程师或代码审查者提供一条快速上手路径
- 任务更适合用引导式序列而非平铺摘要来呈现

示例：
- 新维护者入职
- 某个服务或包的架构导览
- 锚定于变更文件的 PR 审查导览
- 展示故障路径的 RCA 导览
- 聚焦于信任边界和关键检查的安全审查导览

## 何时不使用

| 不使用 code-tour 的情况 | 替代方案 |
| --- | --- |
| 在聊天中一次性解释已经足够 | 直接回答 |
| 用户需要的是文档，而非 `.tour` 产物 | `documentation-lookup` 或仓库文档编辑 |
| 任务是实现或重构 | 执行实现工作 |
| 任务是广泛的代码库入职，无需导览产物 | `codebase-onboarding` |

## 工作流程

### 1. 探索

在编写任何内容之前先探索仓库：
- README 和包/应用入口点
- 目录结构
- 相关配置文件
- 如果是 PR 聚焦的导览，则查看变更文件

在理解代码结构之前不要开始编写步骤。

### 2. 推断读者

根据请求决定用户角色和深度。

| 请求形式 | 用户角色 | 建议步骤数 |
| --- | --- | --- |
| "入职"、"新成员" | `new-joiner` | 9-13 步 |
| "快速导览"、"大致了解" | `vibecoder` | 5-8 步 |
| "架构" | `architect` | 14-18 步 |
| "导览这个 PR" | `pr-reviewer` | 7-11 步 |
| "为什么出问题了" | `rca-investigator` | 7-11 步 |
| "安全审查" | `security-reviewer` | 7-11 步 |
| "解释这个功能如何工作" | `feature-explainer` | 7-11 步 |
| "调试这条路径" | `bug-fixer` | 7-11 步 |

### 3. 读取并验证锚点

每个文件路径和行号锚点必须是真实的：
- 确认文件存在
- 确认行号在有效范围内
- 如果使用选区，验证精确的代码块
- 如果文件易变，优先使用基于模式的锚点

绝不猜测行号。

### 4. 编写 `.tour` 文件

写入：

```text
.tours/<persona>-<focus>.tour
```

保持路径具有确定性且易于阅读。

### 5. 验证

完成前检查：
- 每个引用路径都存在
- 每个行号或选区都有效
- 第一步锚定到真实的文件或目录
- 导览讲述了一个连贯的故事，而非列举文件

## 步骤类型

### 内容步骤

少用，通常只用于结尾步骤：

```json
{ "title": "后续步骤", "description": "你现在可以端到端地追踪整个请求路径了。" }
```

不要将第一步设为纯内容步骤。

### 目录步骤

用于引导读者了解某个模块：

```json
{ "directory": "src/services", "title": "服务层", "description": "核心编排逻辑在这里。" }
```

### 文件 + 行号步骤

这是默认的步骤类型：

```json
{ "file": "src/auth/middleware.ts", "line": 42, "title": "认证门控", "description": "每个受保护的请求都会先经过这里。" }
```

### 选区步骤

当某个代码块比整个文件更重要时使用：

```json
{
  "file": "src/core/pipeline.ts",
  "selection": {
    "start": { "line": 15, "character": 0 },
    "end": { "line": 34, "character": 0 }
  },
  "title": "请求管道",
  "description": "这个代码块连接了验证、认证和下游执行。"
}
```

### 模式步骤

当精确行号可能会漂移时使用：

```json
{ "file": "src/app.ts", "pattern": "export default class App", "title": "应用入口" }
```

### URI 步骤

在有帮助时用于 PR、issue 或文档：

```json
{ "uri": "https://github.com/org/repo/pull/456", "title": "该 PR" }
```

## 写作规则：SMIG

每个描述都应回答：
- **情境（Situation）**：读者正在看什么
- **机制（Mechanism）**：它是如何工作的
- **影响（Implication）**：对当前读者角色为何重要
- **陷阱（Gotcha）**：聪明的读者可能会忽略的事情

保持描述简洁、具体，并基于真实代码。

## 叙事结构

除非任务明显需要不同结构，否则使用此弧线：
1. 定向
2. 模块地图
3. 核心执行路径
4. 边界情况或陷阱
5. 收尾 / 下一步行动

导览应像一条路径，而不是一份清单。

## 示例

```json
{
  "$schema": "https://aka.ms/codetour-schema",
  "title": "API 服务导览",
  "description": "支付服务请求路径导览。",
  "ref": "main",
  "steps": [
    {
      "directory": "src",
      "title": "源代码根目录",
      "description": "服务的所有运行时代码从这里开始。"
    },
    {
      "file": "src/server.ts",
      "line": 12,
      "title": "入口点",
      "description": "服务器在这里启动，在任何路由被触达前连接中间件。"
    },
    {
      "file": "src/routes/payments.ts",
      "line": 8,
      "title": "支付路由",
      "description": "每个支付请求在进入服务逻辑之前都通过这个路由器。"
    },
    {
      "title": "后续步骤",
      "description": "主要锚点就位后，你现在可以端到端地追踪任何支付请求了。"
    }
  ]
}
```

## 反模式

| 反模式 | 修复方案 |
| --- | --- |
| 平铺文件列表 | 讲述步骤间有依赖关系的故事 |
| 泛泛的描述 | 说明具体的代码路径或模式 |
| 猜测锚点 | 先验证每个文件和行号 |
| 快速导览步骤太多 | 果断裁剪 |
| 第一步是纯内容步骤 | 将第一步锚定到真实文件或目录 |
| 角色不匹配 | 为真实读者编写，而不是泛化的工程师 |

## 最佳实践

- 步骤数量与仓库规模和角色深度成比例
- 使用目录步骤定向，使用文件步骤展示实质内容
- PR 导览优先覆盖变更文件
- 对于 monorepo，聚焦于相关包而不是导览所有内容
- 以读者现在能做什么结尾，而不是做总结回顾
