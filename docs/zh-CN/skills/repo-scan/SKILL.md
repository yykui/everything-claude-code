# repo-scan

> 每个生态系统都有自己的依赖管理器，但没有任何工具能跨越 C++、Android、iOS 和 Web 来告诉你：有多少代码是你自己的，哪些是第三方的，哪些是死重。

## 何时使用

- 接管大型遗留代码库时需要结构性概览
- 大规模重构前——识别核心内容、重复内容和废弃内容
- 审计直接嵌入源码（未在包管理器中声明）的第三方依赖
- 为单体仓库重组准备架构决策记录

## 安装

```bash
# Fetch only the pinned commit for reproducibility
mkdir -p ~/.claude/skills/repo-scan
git init repo-scan
cd repo-scan
git remote add origin https://github.com/haibindev/repo-scan.git
git fetch --depth 1 origin 2742664
git checkout --detach FETCH_HEAD
cp -r . ~/.claude/skills/repo-scan
```

> 安装任何代理技能之前，请先审查源代码。

## 核心能力

| 能力 | 描述 |
|---|---|
| **跨栈扫描** | 一次扫描覆盖 C/C++、Java/Android、iOS（OC/Swift）、Web（TS/JS/Vue） |
| **文件分类** | 每个文件标记为项目代码、第三方代码或构建产物 |
| **库检测** | 识别 50+ 已知库（FFmpeg、Boost、OpenSSL 等）并提取版本 |
| **四级评定** | 核心资产 / 提取合并 / 重建 / 废弃 |
| **HTML 报告** | 带钻取导航的交互式深色主题页面 |
| **单体仓库支持** | 带摘要和子项目报告的层级化扫描 |

## 分析深度级别

| 级别 | 读取文件数 | 适用场景 |
|---|---|---|
| `fast` | 每模块 1-2 个 | 快速盘点超大目录 |
| `standard` | 每模块 2-5 个 | 含完整依赖和架构检查的默认审计 |
| `deep` | 每模块 5-10 个 | 额外增加线程安全、内存管理、API 一致性检查 |
| `full` | 所有文件 | 合并前的全面评审 |

## 工作原理

1. **分类仓库界面**：枚举文件，然后将每个文件标记为项目代码、嵌入的第三方代码或构建产物。
2. **检测嵌入的库**：检查目录名、头文件、许可证文件和版本标记，以识别打包的依赖项及其可能的版本。
3. **对每个模块评分**：按模块或子系统对文件分组，然后根据所有权、重复程度和维护成本分配四种评定之一。
4. **突出结构性风险**：指出废弃产物、重复包装器、过时的外部引入代码，以及应被提取、重建或废弃的模块。
5. **生成报告**：返回简洁摘要以及带有模块钻取功能的交互式 HTML 输出，便于异步评审审计结果。

## 示例

在一个包含 50,000 个文件的 C++ 单体仓库中：
- 发现仍在生产中使用的 FFmpeg 2.x（2015 年版本）
- 发现同一 SDK 包装器重复了 3 次
- 识别出 636 MB 已提交的 Debug/ipch/obj 构建产物
- 分类结果：3 MB 项目代码 vs 596 MB 第三方代码

## 最佳实践

- 首次审计从 `standard` 深度开始
- 对于有 100+ 模块的单体仓库，使用 `fast` 快速盘点
- 对标记为需要重构的模块逐步运行 `deep`
- 查看跨模块分析以检测子项目间的重复
