# 医疗 EMR 开发模式

构建电子病历（EMR）和电子健康档案（EHR）系统的模式。优先考虑患者安全、临床准确性和医生效率。

## 何时使用

- 构建患者就诊工作流（主诉、检查、诊断、处方）
- 实现临床病历记录（结构化 + 自由文本 + 语音转文字）
- 设计带有药物相互作用检查的处方/药物模块
- 集成临床决策支持系统（CDSS）
- 构建带有参考范围高亮的实验室结果展示
- 实现临床数据的审计跟踪
- 为临床数据录入设计医疗无障碍 UI

## 工作原理

### 患者安全优先

每个设计决策都必须评估："这会伤害患者吗？"

- 药物相互作用必须发出警报，不能静默通过
- 异常实验室值必须进行视觉标记
- 危急生命体征必须触发升级工作流
- 所有临床数据修改都必须有审计跟踪

### 单页就诊流程

临床就诊应在单页上垂直展开——无需切换标签页：

```
Patient Header (sticky — always visible)
├── Demographics, allergies, active medications
│
Encounter Flow (vertical scroll)
├── 1. Chief Complaint (structured templates + free text)
├── 2. History of Present Illness
├── 3. Physical Examination (system-wise)
├── 4. Vitals (auto-trigger clinical scoring)
├── 5. Diagnosis (ICD-10/SNOMED search)
├── 6. Medications (drug DB + interaction check)
├── 7. Investigations (lab/radiology orders)
├── 8. Plan & Follow-up
└── 9. Sign / Lock / Print
```

### 智能模板系统

```typescript
interface ClinicalTemplate {
  id: string;
  name: string;             // e.g., "Chest Pain"
  chips: string[];          // clickable symptom chips
  requiredFields: string[]; // mandatory data points
  redFlags: string[];       // triggers non-dismissable alert
  icdSuggestions: string[]; // pre-mapped diagnosis codes
}
```

任何模板中的红色标志必须触发可见的、不可关闭的警报——而非浮动通知（toast）。

### 药物安全模式

```
用户选择药物
  → 检查当前用药的相互作用
  → 检查本次就诊用药的相互作用
  → 检查患者过敏史
  → 根据体重/年龄/肾功能验证剂量
  → 如果为严重相互作用：完全阻止开药
  → 临床医生必须记录覆盖原因才能通过阻止
  → 如果为重大相互作用：显示警告，要求确认
  → 将所有警报和覆盖原因记录在审计跟踪中
```

严重相互作用**默认阻止开药**。临床医生必须明确覆盖并记录在审计跟踪中的理由。系统绝不静默允许严重相互作用。

### 锁定就诊记录模式

临床就诊记录一旦签署：
- 不允许编辑——只能添加附录（单独的关联记录）
- 原始记录和附录都出现在患者时间轴中
- 审计跟踪记录签署人、时间及任何附录记录

### 临床数据 UI 模式

**生命体征展示：** 当前值含正常范围高亮（绿/黄/红），与上次的趋势箭头，自动计算临床评分（NEWS2、qSOFA），内联升级指导。

**实验室结果展示：** 正常范围高亮，与上次值对比，危急值带不可关闭的警报，采集/分析时间戳，待处理医嘱及预计周转时间。

**处方 PDF：** 一键生成，包含患者人口统计学信息、过敏史、诊断、药物详情（通用名 + 商品名、剂量、给药途径、频次、疗程），以及临床医生签名栏。

### 医疗 UI 无障碍要求

医疗 UI 的要求比一般 Web 应用更为严格：
- 最低 4.5:1 对比度（WCAG AA）——临床医生在不同光线条件下工作
- 大触摸目标（最小 44x44px）——适用于戴手套或紧急交互
- 键盘导航——适用于快速录入数据的高级用户
- 非纯色指示——始终将颜色与文字/图标配合使用（色觉障碍的临床医生）
- 所有表单字段有屏幕阅读器标签
- 临床警报不能自动关闭——临床医生必须主动确认

### 反模式

- 将临床数据存储在浏览器 localStorage
- 药物相互作用检查静默失败
- 危急临床警报使用可关闭的浮动通知
- 基于标签页的就诊 UI 碎片化临床工作流
- 允许编辑已签署/锁定的就诊记录
- 显示没有审计跟踪的临床数据
- 对临床数据结构使用 `any` 类型

## 示例

### 示例 1：患者就诊流程

```
医生为患者 #4521 开始就诊
  → 顶部固定栏显示："Rajesh M，58 岁男性，过敏：青霉素，当前用药：二甲双胍 500mg"
  → 主诉：选择"胸痛"模板
    → 点击症状标签：" 胸骨下"、"放射到左臂"、"压榨感"
    → 红色标志"胸骨下压榨性疼痛"触发不可关闭的警报
  → 检查：心血管系统——"S1 S2 正常，无杂音"
  → 生命体征：心率 110，血压 90/60，SpO2 94%
    → NEWS2 自动计算：得分 8，风险高，显示升级警报
  → 诊断：搜索"ACS" → 选择 ICD-10 I21.9
  → 药物：选择阿司匹林 300mg
    → CDSS 检查与二甲双胍的相互作用：无相互作用
  → 签署就诊记录 → 锁定，此后只能添加附录
```

### 示例 2：药物安全工作流

```
医生为患者 #4521 开具华法林
  → CDSS 检测到：华法林 + 阿司匹林 = 严重相互作用
  → UI：红色不可关闭的模态框阻止开药
  → 医生点击"有原因地覆盖"
  → 输入："获益大于风险——已监测 INR 方案"
  → 覆盖原因 + 警报存储在审计跟踪中
  → 处方继续开具，并附有文档化的覆盖记录
```

### 示例 3：锁定就诊记录 + 附录

```
就诊记录 #E-2024-0891 由 Shah 医生于 14:30 签署
  → 所有字段锁定——不显示编辑按钮
  → 可用"添加附录"按钮
  → Shah 医生点击附录，添加："收到实验室结果——肌钙蛋白升高"
  → 新记录 E-2024-0891-A1 关联到原始记录
  → 时间轴显示两者：原始就诊记录 + 带时间戳的附录
```
