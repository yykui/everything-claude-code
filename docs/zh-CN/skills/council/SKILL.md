# Council

为模糊决策召集四位顾问：
- 当前上下文中的 Claude 视角
- 怀疑者（Skeptic）子代理
- 务实者（Pragmatist）子代理
- 批评者（Critic）子代理

这适用于**模糊情况下的决策制定**，而非代码审查、实现规划或架构设计。

## 何时使用

在以下情况使用 council：
- 一个决策有多条可信路径，没有明显的优胜者
- 需要明确地呈现权衡取舍
- 用户要求第二意见、异议或多角度观点
- 对话锚定是真实风险时
- 是否执行的判断需要对抗性挑战时

示例：
- monorepo 与 polyrepo
- 现在发布还是等待打磨
- 功能开关还是全量上线
- 简化范围还是保持战略广度

## 何时不使用

| 不使用 council 的情况 | 替代方案 |
| --- | --- |
| 验证输出是否正确 | `santa-method` |
| 将功能拆分为实现步骤 | `planner` |
| 设计系统架构 | `architect` |
| 审查代码的 Bug 或安全问题 | `code-reviewer` 或 `santa-method` |
| 直接的事实性问题 | 直接回答 |
| 明显的执行任务 | 直接执行 |

## 角色

| 声部 | 视角 |
| --- | --- |
| 架构者（Architect） | 正确性、可维护性、长期影响 |
| 怀疑者（Skeptic） | 前提挑战、简化、假设破除 |
| 务实者（Pragmatist） | 交付速度、用户影响、运营现实 |
| 批评者（Critic） | 边界情况、下行风险、失败模式 |

三个外部声部应作为全新的子代理启动，**只携带问题和相关上下文**，而不是完整的持续对话。这是防止锚定的机制。

## 工作流程

### 1. 提炼真正的问题

将决策简化为一个明确的提示：
- 我们在决定什么？
- 什么约束条件重要？
- 什么算成功？

如果问题模糊，在召集议会前先提一个澄清性问题。

### 2. 只收集必要的上下文

如果决策与具体代码库相关：
- 收集相关文件、代码片段、issue 文本或指标
- 保持简洁
- 只包含做决策所必要的上下文

如果决策是战略性/通用性的：
- 除非代码片段对答案有实质影响，否则跳过

### 3. 先形成架构者立场

在阅读其他声部之前，写下：
- 你的初始立场
- 支持它的三个最有力的理由
- 你首选路径中的主要风险

先做这一步，这样综合分析就不会简单地映射外部声部。

### 4. 并行启动三个独立声部

每个子代理获得：
- 决策问题
- 必要时的精简上下文
- 严格的角色定义
- 不必要的对话历史

提示结构：

```text
你是四声部决策议会中的[角色]。

问题：
[决策问题]

上下文：
[只有相关片段或约束条件]

请回应：
1. 立场——1-2 句话
2. 推理——3 个简洁的要点
3. 风险——你建议中的最大风险
4. 惊喜——其他声部可能忽略的一件事

直接表达。不要含糊其辞。保持在 300 字以内。
```

角色重点：
- 怀疑者：挑战框架、质疑假设、提出最简单的可信替代方案
- 务实者：优化速度、简洁性和真实世界的可执行性
- 批评者：呈现下行风险、边界情况和计划可能失败的原因

### 5. 带有偏差防护地进行综合

你既是参与者也是综合者，因此使用以下规则：
- 不要在不解释原因的情况下驳回外部观点
- 如果外部声部改变了你的建议，明确说出来
- 始终包含最强的异见，即使你拒绝它
- 如果两个声部与你的初始立场相对，将其视为真实信号
- 在给出最终结论之前，保持原始立场可见

### 6. 呈现简洁的最终结论

使用以下输出结构：

```markdown
## Council：[简短决策标题]

**架构者：** [1-2 句话的立场]
[1 行说明原因]

**怀疑者：** [1-2 句话的立场]
[1 行说明原因]

**务实者：** [1-2 句话的立场]
[1 行说明原因]

**批评者：** [1-2 句话的立场]
[1 行说明原因]

### 最终结论
- **共识：** [各方一致之处]
- **最强异见：** [最重要的分歧]
- **前提检验：** [怀疑者是否挑战了问题本身？]
- **建议：** [综合后的路径]
```

保持在手机屏幕上可扫读。

## 持久化规则

**不要**从本技能向 `~/.claude/notes` 或其他隐藏路径写入临时笔记。

如果议会实质性地改变了建议：
- 使用 `knowledge-ops` 将经验教训存储在正确的持久位置
- 或者使用 `/save-session` 如果结果属于会话记忆
- 或者如果决策改变了活跃的执行事实，直接更新相关 GitHub / Linear issue

只在决策改变了实质性内容时才持久化。

## 多轮跟进

默认一轮。

如果用户想要更多轮次：
- 保持新问题聚焦
- 只在必要时包含之前的结论
- 尽量保持怀疑者的干净程度以保留防锚定价值

## 反模式

- 将 council 用于代码审查
- 在任务仅是实现工作时使用 council
- 将完整的对话记录传给子代理
- 在最终结论中隐藏分歧
- 不论重要性如何，将每个决策都持久化为笔记

## 相关技能

- `santa-method`——对抗性验证
- `knowledge-ops`——正确地持久化持久决策差异
- `search-first`——在议会召开前收集外部参考资料
- `architecture-decision-records`——当决策成为长期系统策略时，将结果正式化

## 示例

问题：

```text
我们应该现在将 ECC 2.0 作为 alpha 版本发布，还是等到控制平面 UI 更完善后再发布？
```

可能的议会构成：
- 架构者强调结构完整性，避免混乱的界面
- 怀疑者质疑 UI 是否真的是关键制约因素
- 务实者询问现在可以在不损害信任的情况下发布什么
- 批评者聚焦于支持负担、预期债务和发布混乱

价值不在于一致性。价值在于在做出选择之前使分歧清晰可见。
