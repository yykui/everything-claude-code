# 议会

召集四位顾问处理模糊决策：
- 上下文中的 Claude 声音
- 一个怀疑者子 agent
- 一个实用主义者子 agent
- 一个批评者子 agent

这是用于**模糊情况下的决策制定**，而非代码审查、实现规划或架构设计。

## 何时使用

在以下情况使用议会：
- 一个决策有多条可信路径且没有明显赢家
- 需要明确地将权衡取舍浮出水面
- 用户要求征求第二意见、异见或多角度看法
- 对话锚定偏差是真实风险
- 通过/否决判断将受益于对抗性挑战

示例：
- 单体仓库 vs 多仓库
- 现在发布 vs 等待打磨
- 功能标志 vs 全量发布
- 简化范围 vs 保留战略广度

## 何时不使用

| 不使用议会的情况 | 替代方案 |
| --- | --- |
| 验证输出是否正确 | `santa-method` |
| 将功能分解为实现步骤 | `planner` |
| 设计系统架构 | `architect` |
| 审查代码中的 Bug 或安全问题 | `code-reviewer` 或 `santa-method` |
| 直接的事实性问题 | 直接回答 |
| 明显的执行任务 | 直接执行任务 |

## 角色

| 声音 | 视角 |
| --- | --- |
| 架构师 | 正确性、可维护性、长期影响 |
| 怀疑者 | 前提挑战、简化、打破假设 |
| 实用主义者 | 发布速度、用户影响、运营现实 |
| 批评者 | 边界情况、下行风险、失败模式 |

三个外部声音应作为全新的子 agent 启动，**只携带问题和相关上下文**，而非完整的进行中对话。这是防止锚定偏差的机制。

## 工作流

### 1. 提炼真正的问题

将决策简化为一个明确的提示：
- 我们在决定什么？
- 哪些约束条件重要？
- 什么算作成功？

若问题模糊，在召集议会之前先问一个澄清性问题。

### 2. 仅收集必要的上下文

若决策与代码库相关：
- 收集相关文件、代码片段、问题文本或指标
- 保持紧凑
- 只包含做出决策所需的上下文

若决策是战略性/通用性的：
- 除非对答案有实质性改变，否则跳过仓库片段

### 3. 首先形成架构师立场

在阅读其他声音之前，写下：
- 你的初始立场
- 支持它的三个最强理由
- 你偏好路径的主要风险

先做这一步，这样综合结果就不会简单地反映外部声音。

### 4. 并行启动三个独立声音

每个子 agent 获得：
- 决策问题
- 必要时的紧凑上下文
- 严格的角色定义
- 无不必要的对话历史

提示格式：

```text
You are the [ROLE] on a four-voice decision council.

Question:
[decision question]

Context:
[only the relevant snippets or constraints]

Respond with:
1. Position — 1-2 sentences
2. Reasoning — 3 concise bullets
3. Risk — biggest risk in your recommendation
4. Surprise — one thing the other voices may miss

Be direct. No hedging. Keep it under 300 words.
```

角色重点：
- 怀疑者：挑战框架，质疑假设，提出最简单可信的替代方案
- 实用主义者：优化速度、简洁性和现实世界中的执行
- 批评者：浮现下行风险、边界情况以及计划可能失败的原因

### 5. 带偏见防护地综合

你既是参与者也是综合者，因此使用以下规则：
- 不要在不解释原因的情况下驳斥外部观点
- 若某个外部声音改变了你的建议，明确说明
- 始终包含最强的异见，即使你拒绝它
- 若两个声音与你的初始立场相左，将其视为真实信号
- 在给出判决之前保持原始立场可见

### 6. 呈现紧凑的判决

使用以下输出格式：

```markdown
## Council: [简短决策标题]

**架构师：** [1-2 句立场]
[1 行说明原因]

**怀疑者：** [1-2 句立场]
[1 行说明原因]

**实用主义者：** [1-2 句立场]
[1 行说明原因]

**批评者：** [1-2 句立场]
[1 行说明原因]

### 判决
- **共识：** [他们在哪里一致]
- **最强异见：** [最重要的分歧]
- **前提检查：** [怀疑者是否质疑了问题本身？]
- **建议：** [综合后的路径]
```

保持在手机屏幕上可扫描。

## 持久化规则

**不要**从此技能向 `~/.claude/notes` 或其他影子路径写入临时笔记。

若议会实质性地改变了建议：
- 使用 `knowledge-ops` 将教训存储到正确的持久位置
- 或若结果属于会话记忆则使用 `/save-session`
- 或若决策改变了活跃执行事实则直接更新相关的 GitHub / Linear Issue

只在决策改变了真实事物时才持久化。

## 多轮跟进

默认为一轮。

若用户希望再进行一轮：
- 将新问题保持聚焦
- 只在必要时包含上一次判决
- 尽量保持怀疑者的清洁以保留防止锚定偏差的价值

## 反模式

- 将议会用于代码审查
- 当任务只是实现工作时使用议会
- 将整个对话记录传给子 agent
- 在最终判决中隐藏分歧
- 无论重要与否都将每个决策持久化为笔记

## 相关技能

- `santa-method` — 对抗性验证
- `knowledge-ops` — 正确持久化持久决策增量
- `search-first` — 若需要在议会前收集外部参考材料
- `architecture-decision-records` — 当决策成为长期系统策略时将结果正式化

## 示例

问题：

```text
Should we ship ECC 2.0 as alpha now, or hold until the control-plane UI is more complete?
```

可能的议会格局：
- 架构师推动结构完整性并避免混乱的界面
- 怀疑者质疑 UI 是否真的是阻塞因素
- 实用主义者询问现在可以在不损害信任的情况下发布什么
- 批评者关注支持负担、预期债务和发布混乱

价值不在于一致。价值在于在选择之前使分歧变得清晰可见。
